# SDD 和 Harness Engineering——Java开发中的 AI 协作编程实践

> 合并版完整演讲稿，预计时长约 105 分钟（含开场结语）。

## 演讲开场：从"氛围编程"到"硬核驾驭"（10分钟）

各位同事，大家好！很高兴能和大家交流 AI 编程的实践经验。

在座的有些同学可能已经用 Copilot 或 Claude 写了大半年的代码，感觉像开了挂；也有不少同学刚开始尝试，正经历"AI 生成的代码看起来对，一跑就报错"的挫败感。

以前我们和 AI 沟通，主要靠"凭感觉"，这种模式现在有个专门的词叫 **Vibe Coding（氛围编程）**。你给 AI 一个模糊的需求，AI 猜着给你一段代码，运气好能跑通，运气不好你就得花几个小时去帮 AI 调试。这种方式在写小 demo 时很快，但在我们复杂的 Java 业务系统中，它往往会带来重构地狱和隐形 Bug。

今天我们要聊的是一套进阶玩法：**SDD（规范驱动开发）** 和 **Harness Engineering（驾驭工程）**。简单说，就是把 AI 从一个"聊天搭子"变成一个"硬核架构师和资深开发"，核心理念是：**"人掌舵，AI 执行" (Humans steer, agents execute)**。

---

## 第一部分：SDD——为 AI 编程画好建筑图纸（30分钟）

什么是 **SDD（规范驱动开发）**？想象你要盖房子。传统方式是边砌砖边想结构，结果拆了重盖。SDD 则是：先画出详细的图纸（规格文档），让图纸成为唯一的真理来源。

### 1. SDD 的三个实施层级

*   **层级一：规范优先 (Spec-first)**：先写规格说明书，再让 AI 辅助写代码。
*   **层级二：规范锚定 (Spec-anchored)**：规范文档永久保存，功能迭代也要先改规范。
*   **层级三：规范即源码 (Spec-as-source)**：这是最激进的，你只负责维护 Markdown 规范，代码完全由 AI 自动生成，你甚至不用看源码。

### 2. 核心工作流：四阶段循环

*   **Specify (定义)**：用自然语言写清楚"做什么"，关注用户价值和验收条件。
*   **Plan (计划)**：明确技术栈。比如我们 Java 项目中，是用 Spring Boot 3 还是 JDK 17？是用 JPA 还是 MyBatis？
*   **Task (任务分解)**：把大需求拆成一个个 AI 可执行、可验证的原子任务。
*   **Implement (实现)**：AI 根据任务清单写代码，你负责验证。

**Java 案例插播：**
比如我们要写一个订单状态变更的接口。在 SDD 模式下，我会先让 AI 写一份 `order-spec.md`。里面定义清楚：输入是什么，幂等性怎么保证，状态机如何跳转。AI 看到这份详细规范后，生成的 Service 层代码准确率会从 60% 直接飙升到 90% 以上。

### ROI 答疑：写 Spec 真的值得吗？

讲到这里，我主动说一个大家可能在想但没说出口的问题：

**写 Spec 本身要花时间，我直接让 AI 写代码不是更快吗？**

这个疑问完全合理。我的答案是：**取决于任务规模**。

| 任务规模 | 建议 | 原因 |
|---------|------|------|
| 改一个字段、加一行日志 | 直接给 AI 做，跳过 Spec | 写 Spec 的时间比任务本身还长 |
| 半天以内的独立功能 | 写一份迷你 Spec（5-10行） | 对齐成本低，避免方向跑偏 |
| 超过一天的复杂功能 | 完整 Spec 流程 | 返工成本远超写 Spec 的成本 |
| 跨多个模块的系统性改动 | Spec + 任务分解 | 没有文档，AI 和你都会迷失 |

关键的判断标准是：**如果这个需求给新同事讲，你需要说超过5分钟，就值得先写 Spec**。

还有一个隐性收益经常被忽视：Spec 写完之后，它就是你的技术文档。以后新人接手、做 Code Review、写测试用例，都能从 Spec 里找到答案。传统开发模式里，这份文档要么写在 Confluence 里很快过时，要么根本不存在。SDD 让文档的维护成本降到最低，因为它直接驱动代码生成。

## Day 1 Setup：明天打开电脑，第一步做什么？

讲到这里，我猜有人在想：这套流程听起来合理，但我们用的是 Claude Code，CLAUDE.md 放在哪？第一次怎么初始化？

我直接把操作步骤说清楚。

### 第一步：安装 Claude Code 并关联项目

```bash
# 全局安装
npm install -g @anthropic-ai/claude-code

# 进入你的 Java 项目根目录
cd /your-project

# 启动，Claude Code 会自动读取当前目录
claude
```

第一次启动，Claude Code 会扫描项目结构。这时候它对你的项目一无所知，相当于刚入职的新同事。接下来要做的，就是给它一份"入职手册"。

### 第二步：创建 CLAUDE.md

在项目根目录新建 `CLAUDE.md`。**不要一次写完**，先从最关键的三件事开始：

```markdown
# 项目上下文

## 技术栈
- Java 17 / Spring Boot 3.x
- MyBatis-Plus（项目中不使用 JPA）
- MySQL 8.0

## 最重要的架构规则
- Controller 只做参数校验和响应组装，不包含业务逻辑
- Service 层不允许直接调用 Repository（必须经过 Domain 层）
- 所有公共 Service 方法必须有 @Transactional 注解

## 代码风格
- 异常处理：业务异常用 BusinessException，不允许吞掉异常
- 日志：使用占位符格式，禁止字符串拼接
- 依赖注入：只用构造器注入，禁止 @Autowired 字段注入
```

三条规则足够开始。之后每次发现 AI 生成了不符合规范的代码，就把对应规范补进去。`CLAUDE.md` 是活的文档，不需要一开始就写完美。

### 第三步：Claude Code 怎么使用 CLAUDE.md

Claude Code 启动后会自动读取项目根目录的 `CLAUDE.md`，不需要你手动粘贴。你可以在对话开始时用这句话确认它已经加载：

```
请用一句话告诉我，根据 CLAUDE.md，这个项目使用什么 ORM 框架？
```

如果它回答正确，说明上下文已加载。如果它说"没有找到相关信息"，检查文件是否在根目录、文件名是否正确。

### 第四步：第一个真实任务，完整走一遍

选一个**本周你手上正好有的、规模在半天以内的需求**。不要专门找一个需求来练习，直接用真实工作。

```
# 在 Claude Code 里输入：

我需要实现 [你的需求]。

在写代码之前，请先输出一份 Spec 文档，包含：
- 输入/输出定义
- 业务规则和边界条件
- 验收条件（可测试的断言）

确认我同意 Spec 之后，再开始写代码。
```

这一句"确认我同意 Spec 之后再开始写代码"非常重要。默认情况下 AI 会直接跳到实现，你需要主动拦截它，先对齐需求再动手。

## 第二部分：Harness Engineering——给烈马套上马具（30分钟）

有了图纸，AI 还是可能会"脱缰"。这时候就需要 **Harness Engineering（驾驭工程）**。

### 1. "马具"的比喻

如果 AI 模型是一匹威力巨大的**烈马**，那 Harness 就是整套**马具**（缰绳、鞍具）。没有马具，你骑上去会摔得很惨；有了马具，你才能引导马的力量奔向终点。

### 2. 驾驭工程的三大支柱

*   **上下文工程 (Context Engineering)**：确保 AI 在正确的时间看到正确的信息。比如在 Java 项目中，通过 `CLAUDE.md` 告诉 AI 我们的包结构、日志规范和 DTO 转换规则。
*   **架构约束 (Architectural Constraints)**：不仅口头要求，还要机械地强制执行。比如写自定义 Linter 规则，AI 生成的代码如果违反了"Service 不能直接调用 Repo"的规定，合并请求直接拦截。
*   **熵管理/自我修复 (Entropy Management)**：随着代码增多，复杂度会失控。我们需要通过调度后台任务，让 AI 定期清理僵尸代码、修复文档过期。

### 我的实际体感

我分享一个自己的实际体感，不是宣传材料里的数字。

在引入 SDD 和 CLAUDE.md 之前，我用 AI 写代码的模式大概是这样：给一个需求 → AI 生成 → 我发现问题 → 补充说明 → AI 修改 → 我再发现问题……平均一个中等复杂度的功能，来回沟通 5 到 8 轮。

引入之后，变成了：先写 Spec → AI 生成 → 跑测试 → 小幅修正 → 完成。通常 2 到 3 轮。

减少的不只是轮次，更重要的是减少了那种"写完发现方向完全跑偏，整段推倒重来"的情况。这种情况以前大概每周会遇到一两次，现在基本消失了。

我没有办法给你一个精确的"效率提升 X%"的数字，因为变量太多。但如果你在日常工作中也有"AI 没理解我的意思"的挫败感，这套方法对症。

## 第三部分：实战进阶——Java 开发者的秘密武器（30分钟）

这部分内容专门针对那 30% 期待提升的同学，以及正在犹豫的同学。

### 1. 强制性 TDD（测试驱动开发）模式

在 AI 编程中，TDD 是质量的终极守门员。我的建议是：**测试通过是任务完成的硬性门槛**。先让 AI 根据 Spec 写单元测试，测试挂了（Red），AI 再去补实现逻辑（Green），最后重构。这样生成的 Java 代码自带 100% 覆盖率，你完全不需要担心重构会坏掉。

### 2. 利用多 Agent 协同（Agent Teams）

不要把 AI 当成一个单点。你可以运行 **"作家/审查员"模式**：
*   **Agent A (Java 开发者)**：负责写逻辑实现。
*   **Agent B (安全审查员)**：专门盯着 SQL 注入、硬编码凭据等安全坑点。
*   **Agent C (QA 自动化专家)**：负责寻找实现逻辑中的边界漏洞。

### 3. Java 项目的特殊技巧

*   **自动版本升级**：利用 Amazon Q 等工具，可以一键将 Java 8 升级到 17，处理那些让人头大的 API 废弃问题。
*   **MCP 插件扩展**：通过 Model Context Protocol，让 AI 直接读取我们的数据库 Schema、Jira 任务单或监控日志，实现真正的数据闭环。

## 给 Team Lead 的单独一节：怎么在团队里落地

前面讲的内容，大部分是个人操作层面的。如果你是 Team Lead，可能在想另一层问题：

**"这套东西怎么变成团队的统一实践？"**

这个问题比个人实践复杂得多，因为你要处理的不只是工具，还有习惯和协作流程。

### 第一个问题：CLAUDE.md 谁来维护？

不要让每个人各自维护自己的 CLAUDE.md，这样很快会产生不一致。

**建议的做法**：

把 `CLAUDE.md` 纳入代码仓库，和代码一起做版本管理。修改 CLAUDE.md 需要走 PR 流程，就像修改架构配置文件一样。这样做有两个好处：第一，规范变更有记录、有审查；第二，所有人用的是同一份上下文，AI 给不同人生成的代码风格一致。

如果项目里有多个子模块，可以采用层级结构：
```
/CLAUDE.md              ← 全局规范（技术栈、架构规则）
/order-service/CLAUDE.md    ← 订单模块专属规范
/payment-service/CLAUDE.md  ← 支付模块专属规范
```

Claude Code 会自动合并读取层级目录里的 CLAUDE.md。

### 第二个问题：Spec 文档放在哪里管理？

建议在仓库里新建一个 `docs/specs/` 目录，按功能模块组织：

```
docs/specs/
  ├── order/
  │   ├── order-status-update.md
  │   └── order-cancellation.md
  └── payment/
      └── refund-process.md
```

**PR 规范变更**：涉及新功能的 PR，必须同时包含对应的 Spec 文档。Spec 文档放在 PR 描述的最上面，Reviewer 先看 Spec 确认业务逻辑正确，再看代码实现。

这个变化会改变 Code Review 的重心：从"代码写得对不对"转向"逻辑设计对不对，代码是否符合设计"。前者是 AI 擅长的，后者才是 Reviewer 的核心价值所在。

### 第三个问题：怎么推行，不引起反弹？

我建议的推行路径：**不要强制全面铺开，先找一个愿意试的人**。

具体来说，分三个阶段：

**第一阶段（第1-2周）：个人试点**
你自己先用，把 CLAUDE.md 写好，跑通完整的 SDD 流程。目的是搞清楚哪些地方在你们项目里会遇到阻力。

**第二阶段（第3-4周）：小组验证**
找 1-2 个有兴趣的同学一起用，收集反馈，完善 CLAUDE.md 和 Spec 模板。这时候要特别注意：不同人用 AI 的习惯不同，你需要把"必须统一的部分"和"可以各自发挥的部分"分清楚。

**第三阶段（第5周起）：团队推广**
把经过验证的 CLAUDE.md 模板、Spec 模板、Prompt 速查表整理成"新人上手包"，作为团队规范推出。这时候有了前两个阶段的案例，推行阻力会小很多。

### 第四个问题：AI 生成的代码，Code Review 怎么做？

这是一个很实际的问题。AI 生成的代码量可能比以前大得多，如果 Review 标准不变，Reviewer 会被淹没。

有几个调整建议：

**自动化承担机械性检查**：代码格式、命名规范、架构约束（ArchUnit）这些，全部交给 CI 自动检查。Review 时不要在这些问题上花时间。

**人工 Review 聚焦三件事**：
1. Spec 和实现是否一致（业务逻辑层面）
2. 边界条件是否覆盖（特别是并发、幂等、异常路径）
3. 测试用例是否真正在验证 Spec 里的验收条件

**引入"Spec Review"前置**：对于较大的需求，在写代码之前，先做一轮 Spec Review。5分钟对齐 Spec，远比事后发现方向错了推倒重来划算。这个习惯建立起来之后，代码层面的 Review 会轻松很多，因为大方向已经在 Spec 阶段确认过了。

## 第四部分：经验教训与避坑指南（10分钟）

这一年多我也踩了不少坑，总结几条给刚开始实践的同学：

*   **警惕"Markdown 怪兽"**：不要为了写规范而写出一大堆过时的文档，规范要保持短小、原子化、可验证。
*   **不要过度微观管理**：明确意图和约束后，给 AI 发挥空间。就像带新人，如果你连每一行怎么写都要教，那还不如自己写。
*   **两轮失败原则**：如果对 AI 的修正超过两次还没过，说明你的提示词有问题或者上下文太乱了。果断 `/clear` 重来。

## 结语：从"码农"到"系统驾驭者"（10分钟）

最后我想对那 10% 还没切换到 AI 编程的同学说：AI 并不是要取代你的编码能力，而是要释放你的**架构设计能力和精准表达能力**。

在 AI 时代，未来的核心竞争力不再是写得一手漂亮的 Java 语法，而是你能否设计出一套坚固的"马具"，指挥一群 AI 智能体精准地构建出复杂的系统。

规范先行，代码自动。让我们一起从"写代码的人"变成"设计系统的人"！

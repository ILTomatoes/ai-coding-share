## 一、设计-实现-测试闭环

### 整体流程

```
设计阶段          实现阶段               测试阶段           复盘阶段
┌──────────┐    ┌──────────────┐     ┌──────────────┐   ┌──────────┐
│ IDE 写    │    │ Plan 模式    │     │ 单元测试      │   │ /retro-  │
│ 设计 MD   │ →  │ → Sub Agent  │ →   │ + Karate     │ → │ spect    │
│ → Claude  │    │ → Agent Team │     │ 业务接口测试  │   │ 三问复盘  │
│ 输出计划   │    │ → Worktree   │     │              │   │          │
└──────────┘    └──────────────┘     └──────────────┘   └──────────┘
```

### 1.1 设计阶段：PRD 质量决定成败

**核心发现**:
- **PRD 清晰** → AI 事半功倍
- **PRD 模糊** → AI 飞速跑偏，无法验收

**实践做法**:
1. 使用 IDE（Lingma IDE）写设计 Markdown
2. Claude Code 输出实施计划
3. 人工审阅计划，确认后启动实现

**设计文档示例**（aip-portal 项目中）:
- 14 个设计文档存放在 `docs/design/`
- 包括 SQL 版本管理 Starter 设计（547 行）
- Karate 接入实践（668 行）
- 消息管理重构设计（283 行）

### 1.2 实现阶段：多种协作模式

根据任务复杂度选择不同模式：

| 任务复杂度 | 推荐模式 | 适用场景 |
|-----------|----------|----------|
| 简单修改 | 直接对话 | 单文件修改、小 bug 修复 |
| 中等功能 | Plan 模式 | 需要探索代码再实现的任务 |
| 复杂功能 | Sub Agent | 需要拆分子任务，控制上下文 |
| 多模块功能 | Agent Team | 多 Agent 并行开发 |
| 多任务并行 | Git Worktree | 物理隔离，多任务同时进行 |

### 1.3 测试阶段

**单元测试**:
- AI 生成 + 人工审查
- 注意维护成本：AI 生成的测试有时过于脆弱

**Karate 测试**（aip-portal 项目实践）:
- **框架**: Karate DSL
- **测试策略**: 三层数据隔离
- **标签体系**: @smoke / @p0 / @p1 / @p2 / @negative
- **环境配置**: 开发禁用 Flyway，生产启用
- **执行**: `mvn test -Dtest=KarateRunner -Dkarate.tags=@smoke`

### 1.4 复盘阶段

使用自定义 `/retrospect` 技能进行三问复盘：
1. 做对了什么？
2. 做错了什么？
3. 如果重来会怎么做？

## 二、Plan 模式实践

### 工作方式

```
用户需求 → Plan 模式启动
              ↓
         探索代码库（Explore Agent）
              ↓
         制定实施计划
              ↓
         用户审阅确认 ← 关键环节
              ↓
         执行实现
```

### 实际使用案例

**案例：aip-portal 多模式交叉搜索权限逻辑**

在搜索权限过滤相关逻辑时：
1. 同时搜索 `!isAdmin` 变量名
2. 搜索中文注释（如"权限"、"角色"）
3. 搜索 `isUserWorkspaceAdmin` 方法名
4. 交叉验证，避免单一模式遗漏

**Plan 模式的价值**:
- 避免了直接动手改导致的遗漏
- 可以在计划阶段标注需要确认的关键信息
- 用户审阅后再执行，减少返工

## 三、Git Worktree 并行开发

### 使用规模

在 aip-portal 项目中创建了 **24 个 worktree**，覆盖不同模块：

```
aip-portal/
├── .claude/worktrees/
│   ├── app/           # 应用管理
│   ├── apply/         # 申请流程
│   ├── assistant/     # 智能助理（最大模块）
│   ├── auth/          # 认证授权
│   ├── bot/           # 机器人
│   ├── channel/       # 渠道管理
│   ├── employee/      # 员工管理
│   ├── fav/           # 收藏功能
│   ├── grant/         # 授权管理
│   ├── mem/           # 内存管理
│   ├── message/       # 消息管理
│   ├── ops/           # 运维管理
│   ├── role/          # 角色管理
│   ├── schedule/      # 调度管理
│   └── ...            # 更多模块
```

### Worktree 工作流

```
主分支 (main)
  ├── worktree/employee → Claude Code 会话 1（独立上下文）
  ├── worktree/message  → Claude Code 会话 2（独立上下文）
  └── worktree/role     → Claude Code 会话 3（独立上下文）
       ↓                ↓                ↓
    独立开发          独立开发          独立开发
       ↓                ↓                ↓
    合并回主分支 ← ← ← ← ← ← ← ← ← ←
```

### 核心优势

1. **物理隔离**: 每个 worktree 有独立的工作目录和文件系统
2. **上下文独立**: 每个 Claude Code 会话有自己的上下文，互不干扰
3. **避免冲突**: 不同模块的开发不会互相影响
4. **并行高效**: 可以同时开启多个会话处理不同模块

## 四、Agent Team 多 Agent 协作

### 团队配置

在"数字员工门户"项目中使用了 **5 人 Agent 团队**：

```
                    ┌──────────────────┐
                    │    Team Lead     │
                    │   （用户/你）     │
                    └────────┬─────────┘
                             │
              ┌──────────────┼──────────────┐
              ↓              ↓              ↓
     ┌────────────┐  ┌────────────┐  ┌────────────┐
     │ 需求分析师  │  │ 表结构设计师│  │ 接口设计师  │
     │ requirements│  │ table-     │  │ api-       │
     │ -analyst   │  │ designer   │  │ designer   │
     └────────────┘  └────────────┘  └────────────┘
              │              │              │
              └──────────────┼──────────────┘
                             ↓
                    ┌──────────────────┐
                    │   计划工程师      │
                    │   plan-engineer  │
                    └────────┬─────────┘
                             ↓
                    ┌──────────────────┐
                    │   评审验收        │
                    │   reviewer       │
                    └──────────────────┘
```

### 工作流程

```
1. 需求确认 → 需求分析师梳理需求
2. 并行设计 → 表结构设计师 + 接口设计师 同时工作
3. 方案评审 → 评审验收角色检查
4. 计划生成 → 计划工程师输出实施计划
5. 开发执行 → 基于 plan 启动编码
```

### 技术实现

- **配置文件**: `~/.claude/teams/digital-employee-team.md`
- **消息机制**: 队列式消息传递（requirements-analyst.json, team-lead.json）
- **实验功能**: `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS: "1"`

## 五、自定义技能系统

### 技能 1：guided-learning-tutor（通用学习引导导师）

**核心理念**: 苏格拉底方法 + 主动验证

**设计特点**:
- **学生分类**: 纯初学者 / 转专业者 / 有经验者 / 专家
- **三级掌握标准**:
  - L1 能解释
  - L2 能应用
  - L3 能教他人
- **五步交互结构**: 初始探索 → 解释 → 动手实践 → 代码审查 → 自适应跟进
- **进度追踪**: 双重更新机制（会话记录 + 总体追踪）

**跨领域验证**: 编程 / 机器学习 / 投资分析 / 语言学习

### 技能 2：retrospect（复盘技能）

**触发方式**: `/retrospect` 命令

**核心流程**:
```
收集上下文（当前任务、代码变更）
    → 三问复盘（做对了/做错了/重来怎么做）
    → 提取可复用经验
    → 保存到 memory/（feedback 类型）
    → 可选：更新 rules/ 或 CLAUDE.md
```

**独特设计**: 将经验"制度化"——不仅仅是记录，而是固化为规则

### 技能 3：commit-and-push / quick-commit

**commit-and-push**: 完整的提交推送流程
- 分析变更 → 生成 commit message → 排除 Claude 文件 → 提交推送

**quick-commit**: 快速提交
- 自动生成符合项目风格的 commit message
- 可选 `--push` 参数

**共同规则**:
- 禁止添加 Co-Authored-By
- 使用中文描述
- 绝不提交敏感文件

## 七、模型选择策略

### "奢侈但有效"的策略

**核心原则**: 不要在低效的循环里死磕

| 场景 | 推荐模型 | 原因 |
|------|----------|------|
| 日常编码 | Sonnet | 性价比最高，速度最快 |
| 系统设计、架构规划 | Opus | 推理能力更强，适合复杂决策 |
| 反复尝试无果时 | 切换到 Opus | 更强的模型可能一次性解决 |
| 简单搜索、快速查询 | Haiku | 速度快，成本低 |

**实际使用**: 通过 cc-switch 工具在不同模型间切换

## 九、工作流最佳实践总结

### 核心原则

1. **探索先行**: 复杂任务先探索代码结构，再做方案
2. **确认再动**: 需求有多种可能时，先确认再实现
3. **小步快跑**: 不要一次做太大，及时验证
4. **复盘驱动**: 每次踩坑都是完善知识体系的机会
5. **上下文管理**: 控制 AI 上下文，不是越多越好

### 不同规模任务的推荐策略

| 任务规模 | 推荐策略 | 时间预估 |
|----------|----------|----------|
| 小 bug 修复 | 直接对话 + 编译验证 | 分钟级 |
| 单功能开发 | Plan 模式 → 实现 → 测试 | 小时级 |
| 多模块功能 | Sub Agent 拆分 → 逐个实现 | 半天级 |
| 大型重构 | Agent Team + Worktree 并行 | 天级 |

### [PPT 3-4] AI 编程的三个常见误区（2:00 - 7:00）

#### [PPT 3] 误区一和误区二

用了 AI 编程工具之后，我发现有三个常见的认知误区，我自己也全都踩过。

**第一个误区：AI 什么都会。** 很多时候我们觉得，把需求丢给 AI，它就应该能搞定。但实际上，AI 只能看到你给它的上下文。就像你让一个新来的同事写代码，但不给他看项目架构文档、编码规范、已有的工具类——他怎么可能写出符合项目要求的代码？AI 也一样。你给它什么信息，它就在这个范围内工作。信息不够，它就自己"发挥"，而 AI 的发挥往往非常自信——但方向可能是错的。

**第二个误区：告诉 AI 就够了。** 这是我自己踩的一个大坑。有一次我让 AI 生成了大量代码，没有充分审查就直接采纳了。结果到了对接联调的时候，大量的返工、大量的调试。这让我学到一个教训：缺少审查的 AI 编程，比手写代码更危险——因为你会有一种"已经完成了"的错觉。

#### [PPT 4] 误区三 + 转折

**第三个误区：一次就能做好。** 很多人（包括我自己）一开始会试图把 AI 的配置做到完美，把所有规则一次性写全。但实际情况是，你不可能在项目开始时就预测 AI 会犯什么错。好的做法是：先让它跑起来，观察它的行为，遇到问题再加约束。

这三条误区有一个共同的根本原因：**我们没有为 AI 搭建一个合适的工作环境。**

### [PPT 5] 核心概念引入：Harness 比喻（7:00 - 10:00）

这里我想引入一个概念，叫 **Harness Engineering**，翻译过来就是"马具工程"。

想象一下：AI 就像一匹千里马，能力很强、速度很快。但如果你直接骑上去，没有缰绳、没有马鞍，它可能跑得很快——但方向不对，甚至会把你摔下来。

**Harness，就是这副缰绳和马鞍。**

缰绳不增加马的力量，但它让力量用在对的方向。同样，我们给 AI 写的配置文件、规则、约束，不会让 AI 变得更聪明，但能让它的输出更符合我们的预期。

所以今天分享的核心不是"怎么用 AI 工具"——这方面大家都有经验。今天分享的是：**怎么为 AI 搭建一个能让它高效工作的环境。** 具体来说，就是知识怎么组织、规则怎么积累、工作流怎么设计。

### [PPT 6] 三层知识架构全景图（10:00 - 18:00）

#### [PPT 6] 架构图展示

现在进入今天最核心的部分：知识体系的设计。

在我的项目中，我发现给 AI 传递知识的最佳方式是分三个层级。让我先展示这张架构图：

```
┌──────────────────────────────────────┐
│  全局层 (~/.claude/CLAUDE.md)         │
│  → "我是谁，我怎么工作"               │
│  → 所有项目共享                       │
├──────────────────────────────────────┤
│  项目层 (CLAUDE.md + rules/)          │
│  → "这个项目怎么做"                   │
│  → 每次对话全部加载                   │
├──────────────────────────────────────┤
│  会话层 (memory/)                     │
│  → "上次踩了什么坑"                   │
│  → 按需检索                          │
└──────────────────────────────────────┘
```

每一层的职责不同、生命周期不同、加载方式也不同。

#### [PPT 7] 全局层详解

**全局层**，放在 `~/.claude/CLAUDE.md` 里，是你跨所有项目的个人偏好。

这里只放两样东西：你所有的项目都通用的规则，和你长期稳定的工作习惯。

比如我放了这些：
- Git 提交规范：用中文的 `type:` 格式，禁止加 Co-Authored-By 署名
- Shell 环境规范：Windows Git Bash 中重定向用 `/dev/null` 而不是 `> nul`（这个是踩了坑才加的，后面会讲）

原则是：**这个文件的内容应该几乎不变。** 如果某个规则只跟某个项目有关，它就不应该出现在这里。

#### [PPT 8] 项目层详解

**项目层**是最重要的层级。它包括项目根目录的 `CLAUDE.md` 和 `.claude/rules/` 目录下的文件。

这里有一个非常关键的特点：**项目层的文件，每次和 AI 对话时都会全部加载。** 也就是说，你写的每一行都在消耗 token。AI 不需要的信息就是噪音。

所以这里有一条核心原则——也是我在实践中得到的最重要的一条：

> **"rules/ 是给 AI 的参考，不是给人看的 wiki。"**

很多人（包括我自己一开始）会把 rules/ 当成一个知识库来维护，什么设计文档、执行计划、甚至数据库 schema 都往里塞。这样做的后果是：AI 每次对话都要阅读几千行的无关内容，效率直线下降，而且容易产生混乱。

#### [PPT 9] 会话层详解

**会话层**，放在 `memory/` 目录下，是跨会话的经验积累。

它和项目层的最大区别是：**不自动加载，按需检索。** 也就是说，AI 会根据当前任务的上下文，自动查找相关的记忆。这意味着你可以更自由地在这里积累经验，不用担心占用每次对话的 token 预算。

我主要在这里保存两类内容：
- **复盘经验**（feedback 类型）：每次完成任务后复盘，把踩过的坑沉淀下来
- **外部资源索引**（reference 类型）：比如 rules 文件的管理规范

#### [PPT 10] 设计原则总结

总结一下三层架构的设计原则：

| 原则 | 说明 |
|------|------|
| **信息不重复** | 同一条规则只存在于一个层级，其他层级引用 |
| **层级边界清晰** | 全局放"我是谁/我怎么工作"，项目放"这个项目怎么做"，会话放"上次踩了什么坑" |
| **Token 有意识** | 项目层每次全部加载，所以必须有存在价值；会话层按需检索，可以更自由 |

这三条原则，在实际使用中可以帮你判断任何一条信息"应该放在哪里"。

### [PPT 11-14] rules/ 配置的演变历程——一个真实项目的故事（18:00 - 26:00）

#### [PPT 11] Phase 1-2：混乱期

现在讲一个真实的故事——我的项目中 rules/ 配置是怎么一步步演变的。这个故事的起伏本身就是最好的经验。

**Phase 1，项目刚启动的时候（3月25-26日）：**

我的 rules/ 目录里只有一个 `CODE-STYLE.md`，819 行。是的，一个文件 819 行。当时想的是"把编码规范写全了，AI 就能写出好代码"。结果就是一个大杂烩——命名规范、注解用法、数据库约定、API 设计全部挤在一起。没有测试规范。AI 读取效率很低。

**Phase 2，功能开发推进后（3月27-31日）：**

这时候开始出问题了。每做一个新功能，我就会往 rules/ 里加文档——API 设计文档、功能设计文档、执行计划。到后来：
- 设计文档到了 1706 行（一个"智能助理"功能的设计）
- 执行计划动辄 1000+ 行
- 还有一个 `completed/` 目录，放着已经完成的旧计划

问题是什么？**每次对话都加载所有文件，包括已经完成的历史计划。** 这就好比你上班时，HR 把过去三年所有已结束的项目计划都堆在你桌上——你找当前需要的信息反而更难了。

#### [PPT 12] Phase 3：关键转折

**Phase 3，4月1日，这是一次关键转折。**

我做了一个重要的决定，确立了 rules/ 的核心原则：

> **rules/ 是给 AI 的参考，不是给人看的 wiki。**

然后做了两件事：

**拆分**：把 819 行的 CODE-STYLE.md 拆成 8 个专题文件——命名、注解、数据库、API 设计、注释日志、VO/DI、日期常量、分页——加一个索引文件。每个文件聚焦一个主题，AI 检索效率大幅提升。

**大清理**：一口气删除了 6278 行。已完成的执行计划、设计文档、可再生的数据库 schema——全部从 rules/ 移走。设计文档放到项目的 `docs/` 目录，那是给人看的。

#### [PPT 13] Phase 4：稳定维护

**Phase 4，精简之后（4月2日至今）：**

进入了稳定维护期。最终的 rules/ 结构是这样的：

```
rules/
├── ARCHITECTURE.md     # 架构全景
├── CODE-STYLE.md       # 规范索引
├── code-style/         # 8个专题文件
│   ├── 01-naming.md
│   ├── 02-annotations.md
│   ├── 03-database.md
│   ├── 04-api-design.md
│   ├── 05-comments-logging.md
│   ├── 06-vo-di.md
│   ├── 07-date-constants.md
│   └── 08-pagination.md
└── testing.md          # 测试规范
```

这个结构出来之后，编码规范文件几乎不需要再改。只有架构变更时才更新 ARCHITECTURE.md。

#### [PPT 14] 演变时间线与经验总结

用一个时间线来回顾整个演变过程：

```
Day 1       Week 1        Week 2        Now
  |           |             |            |
819行       +设计文档      -6278行       稳定
单文件      混乱膨胀       大精简重构     维护
```

这个故事最重要的结论是：**好的结构不是一步到位的，而是"膨胀 → 识别问题 → 大清理 → 稳定"的自然循环。先有再优化。** 早期大胆堆内容，有积累后再精简，比一开始追求完美更实际。

### [PPT 15] 记忆系统——把踩过的坑变成制度（26:00 - 30:00）

最后介绍会话层中的一个重要机制：**记忆系统**。

我设计了一个 `/retrospect` 命令——每次完成一个任务后，我会触发复盘。复盘会问三个问题：
1. 做对了什么？
2. 做错了什么？
3. 如果重新做会怎么做？

然后有一个过滤标准：一次性的技术细节不要、能从代码推导的信息不要。要保存的是：**踩坑原因和预防方法**、**工作流改进方向**。

每条记忆都有统一格式：**规则 + Why（为什么）+ How to apply（何时用）**。

举一个真实例子：

> **做错了**：SQL 迁移版本号命名错误（用了 V2.2.14 而不是 V2.2.13）。
> **Why**：没有在开始前确认当前开发版本号，依赖了错误假设。
> **How to apply**：涉及版本号命名时，先查看 `resources/db/migration/postgresql/` 下已有的文件。

这种记忆的价值在于：同样的错误不会犯第二次。而不复盘，"下次注意"就是一句空话。

### [PPT 16-17] 从"直接开干"到"先规划再做"（30:00 - 35:00）

#### [PPT 16] 早期 vs 后来对比

有了知识体系，接下来讲工作流——也就是你和 AI 日常协作的流程。

我早期的工作方式是"直接开干"——告诉 AI 要做什么，然后开始写代码。Git 记录清楚地反映了这一点：连续的 fix、fix、fix，说明一直在边做边改。

后来逐步形成了一个"先规划再做"的工作流。

#### [PPT 17] 推荐工作流

推荐的工作流分四步：

1. **Plan 阶段**：先让 AI 探索现有代码结构，识别可复用的组件和模式
2. **确认阶段**：对关键不确定信息先确认——版本号、字段来源、依赖版本等。这些信息如果猜错了，后面全是返工
3. **实现阶段**：确认方案后才进入编码
4. **验证阶段**：实现后运行编译验证

这个流程看似比"直接开干"多了几个步骤，但它大大减少了返工。我的 git 记录从连续的 fix 模式，变成了清晰的 Plan → 实现 → 验证模式。

---

### [PPT 18] PRD 质量决定成败（35:00 - 39:00）

在工作流中有一个环节特别重要，值得单独拿出来说：**需求文档的质量。**

这是我实践中感受最深的一点：

> "原型/PRD 的质量高，AI 编程就事半功倍。反之，如果 PRD 本身模糊不清甚至自相矛盾，AI 编码大概率越走越偏、飞速跑偏，没法验收、无法判断对错。"

这不是 AI 的问题，这是输入质量决定输出质量的基本规律。

**好的 PRD** 长什么样？明确的输入输出、清晰的边界条件、具体的异常处理要求。

**差的 PRD** 长什么样？"做一个类似 XX 的功能"——这种模糊描述，AI 会按照它训练数据中最常见的模式来做，但那可能完全不是你想要的。

所以我的建议是：在让 AI 写代码之前，先花时间把需求理清楚。这个时间绝对不会白花。

### [PPT 19] 小批量推进 + Hook 自动化（39:00 - 44:00）

接下来是一个我用血泪换来的教训。

前面提到过：有一次我让 AI 生成了大量代码，没有充分验证就直接采纳了。等到联调对接的时候，发现大量问题，返工调试花的时间比从头写还多。

**教训一：小批量推进验收，逐渐加快节奏。** 不要一口气让 AI 写几百行代码然后才检查。每完成一个小模块就编译验证，确认方向正确再继续。

**教训二：不要妄图一次性把刀磨好，遇到问题再磨刀。** 不要想着先把所有规则都写全了再开始。先跑起来，遇到问题再补充规则。

为了解决"忘记验证"的问题，我后来配置了 **Hook 自动化**。在 Claude Code 的 settings.json 中配了一个 PostToolUse 钩子：每次修改 Java 文件后，自动执行 `mvn compile`。这样 AI 写完代码，编译结果直接反馈回来，不需要我手动提醒它去做编译检查。

```json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit|Write",
      "command": "cd /path/to/project && mvn compile -q -DskipTests 2>&1 | tail -5"
    }]
  }
}
```

这种自动化不仅减少了手动操作，更重要的是**把验证变成了流程的一部分**，不会遗漏。

### [PPT 20-23] 从错误中提炼规则——Harness 生长循环（44:00 - 50:00）

#### [PPT 20] 踩坑沉淀表格

最后一部分方法论，讲一个核心的反馈机制：**从错误中提炼规则。**

在我的项目中，每踩一个坑，我都会分析根因，然后沉淀到对应的知识层级。看这个表格：

| 踩坑场景 | 沉淀结果 |
|----------|----------|
| Git Bash 中 `> nul` 创建了垃圾文件 | 全局 CLAUDE.md 的 Shell 重定向规范 |
| Claude 默认在 git commit 加署名 | 全局 CLAUDE.md 的 Git 提交规范 |
| SQL 迁移版本号命名错误 | memory feedback 记录 |
| 对权限控制的理解出现偏差 | memory feedback + ARCHITECTURE.md 更新 |

注意，不同类型的错误沉淀到不同的层级。这是有讲究的。

#### [PPT 21] Harness 生长循环

这个过程其实就是一个循环，我把它叫做 **Harness 生长循环**：

```
让 AI 做任务
    ↓
观察 AI 的输出
    ↓
发现不符合预期的地方
    ↓
分析根因（缺上下文？缺规则？还是偶发？）
    ↓
添加最小约束
    ↓
验证效果
    ↓
回到"让 AI 做任务"，循环往复
```

这个循环的关键是：**每次只添加解决当前问题的最小约束。** 这不是偷懒，这是工程纪律。每一条规则都应该有一个具体的"事故"作为来源，否则它就是猜测。

#### [PPT 22] 错误分类框架

不是所有错误都需要加规则。这里有一个简单的分类框架：

| 错误类型 | 频率 | 应对策略 |
|---------|------|---------|
| 上下文缺失 | 几乎每次 | 在 CLAUDE.md 写明技术栈 |
| 规则不明 | 前几次 | 犯过一次就加规则 |
| 架构偏离 | 项目变大后 | 分层文档 + 自动检查 |
| **偶发错误** | **偶尔** | **直接修复，不要加规则** |

#### [PPT 23] 黄金法则

这里有一条黄金法则：

> **如果一个错误只发生一次，且不太可能再发生，直接修复它，不要加规则。**

为什么？因为每加一条规则，就增加了一份维护成本和 AI 的阅读负担。过多的规则带来的阅读成本，比偶发错误本身造成的问题更大。

### [PPT 24] 演示 A：从零搭建 Spring Boot 项目的 Harness（50:00 - 60:00）

现在进入实战演示环节。我会展示一个 Spring Boot 项目从"裸奔"到搭建 Harness 的完整过程。

#### 视频 1：没有 CLAUDE.md 时的 AI 输出（~2 分钟）

> 播放视频

这是一个刚从 Spring Initializr 生成的项目，什么配置都没有。我让 AI 帮我添加一个"创建任务"的接口。

大家看到了什么？AI 把业务逻辑直接写在了 Controller 里，直接操作 Repository。没有 Service 层，没有 DTO，没有统一的异常处理。代码能跑，但这不是我们期望的架构。

这就是"没有 Harness 时 AI 会做什么"——它走最短路径。

#### 视频 3：逐步添加约束的生长过程（~4 分钟）

> 播放视频

接下来演示 Harness 的"生长"过程。

第一次犯错：业务逻辑在 Controller → 添加分层架构规则
第二次犯错：直接返回 Entity → 添加 DTO 分离规则
第三次犯错：异常处理风格不一致 → 添加统一异常处理规则

每次添加一条规则，CLAUDE.md 从 3 行长到 9 行、15 行、最终 30 行左右。每一条规则的来源都是一个具体的"事故"。

这就是我前面讲的 Harness 生长循环的实战版：**让 AI 做 → 观察 → 发现问题 → 添加最小约束 → 验证 → 循环。**

### [PPT 25] 演示 B：多任务并行与闭环工作流（60:00 - 68:00）

#### 视频 4：worktree 多任务并行（~3 分钟）

> 播放视频（配合 `my-ideas/images/多个worktree并行.png` 截图）

接下来展示一个进阶用法：使用 git worktree 实现多任务并行。

在实际开发中，你可能同时有多个需求在推进。传统做法是频繁切换分支，但 AI 的上下文会在切换中丢失。worktree 的做法是：每个任务一个独立的工作目录，各自有独立的 AI 会话，互不干扰。

大家看截图：这两个终端窗口分别是两个 worktree，一个在做功能 A，一个在做功能 B。两个 AI 实例同时工作，互不影响。

#### 视频 5：设计-开发-测试闭环（~3 分钟）

> 播放视频

最后展示一个完整的闭环工作流。

第一步：用 AI 辅助写设计文档。这个阶段用 Plan 模式，让 AI 先探索代码结构，再输出设计方案。

第二步：Claude Code 执行实现。基于设计方案编码，有明确的检查点。

第三步：测试验证。对于 Java 项目，我用 Karate 做接口级别的集成测试，比单元测试更贴近业务验证。

整个闭环的关键是：**每个环节有明确的产出物，下一个环节基于上一个环节的产出物工作。** 避免了"一口气做完发现方向错了"的问题。

### [PPT 26] 演示 C：复盘与记忆积累（68:00 - 73:00）

#### 视频 6：/retrospect 复盘（~3 分钟）

> 播放视频

最后一个演示：复盘机制。

在完成一个小任务后，我输入 `/retrospect` 触发复盘。大家看：AI 自动回顾了这次任务的过程，输出了三个部分：做对了什么、做错了什么、如果重新做会怎么做。

然后它会过滤出值得保存的经验，按照"规则 + Why + How to apply"的格式，写入 memory 目录。

下次开新会话时，AI 会自动检索相关的记忆。比如我再做一个涉及数据库迁移的任务，它会自动调出之前关于"版本号命名"的记忆，提醒我先确认版本号。

**这就是"经验制度化"——把"下次注意"从一句空话变成可执行的知识记录。**

### [PPT 28-29] Skills 工作流链（75:00 - 79:00）

#### [PPT 28] 完整工作流流程图

前面讲了基本的"先规划再做"工作流。当你更熟练之后，可以用更完整的工作流链：

```
需求提出
    ↓
brainstorming     ← 探索需求和设计方案
    ↓
writing-plans     ← 编写实施计划
    ↓
executing-plans   ← 执行计划（带审查检查点）
    ↓
verification      ← 完成前验证（运行测试、编译）
    ↓
code-review       ← 代码审查
    ↓
retrospect        ← 复盘，保存经验
```

每个环节都有明确的检查标准，避免跳步导致的返工。

#### [PPT 29] 工具组合

除了 Claude Code 本身，还有一些配套工具：

- **playwright-mcp**：可以用来自动化浏览器操作。我之前用它来做系统调研——让 AI 自动打开一个系统，操作页面，然后输出操作文档。也用它来编写用户手册，AI 直接看页面截图就能描述功能。
- **cc-switch**：一个模型切换工具，方便在不同模型之间切换。

### [PPT 30] 模型选择策略（79:00 - 82:00）

关于模型选择，我的经验是：

**设计阶段用强模型，编码实现阶段可以用性价比更高的模型。**

我自己有一个做法：设计文档用 Opus 来写——它是目前最强的模型，对架构设计、需求分析的理解力最好。然后编码实现可以切换到其他性价比更高的模型。

有人说这样做"奢侈"。但我发现了一个事实：**如果和 AI 交互了很多次都得不到好方案，直接换用 Opus 反而更省时间。** 这就是"奢侈但有效"。

关键不是省钱，而是**把每个模型用在它最擅长的环节。**

#### [PPT 32] 多任务并行架构

特别是多任务并行，对于大型功能开发、多模块重构很有价值：

- **worktree**：每个任务一个独立工作目录，互不干扰
- **agent teams**：多个 AI agent 协作，一个负责探索代码、一个负责写计划、一个负责写代码

这已经是比较进阶的用法了，但知道有这个能力，在某些场景下会非常有用。

### [PPT 33] 六大关键经验回顾（85:00 - 87:00）

最后，用六条经验来总结今天的内容：

1. **rules/ 是 token 预算管理** — 每个文件都要有存在价值，AI 不需要的信息就是噪音
2. **记忆系统是经验制度化** — 不复盘等于白做，不保存等于白复盘
3. **三层架构各司其职** — 全局放偏好，项目放规范，会话放经验，不要把信息放错层级
4. **先有再优化** — 早期大胆堆内容，有积累后再精简，比一开始追求完美更实际
5. **从错误中提炼规则** — 每次踩坑都是完善知识体系的机会
6. **文档有生命周期** — 创建→迭代→稳定→过时（更新或删除），不让过期文档占用 token

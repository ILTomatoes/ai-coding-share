## 一、三大翻车案例

### 翻车 1：SQL 迁移版本号错误

**日期**: 2026-04-08
**复盘文件**: `feedback-retrospect-20260408b.md`

#### 事故经过
在 aip-portal 项目中创建 SQL 迁移脚本时，版本号命名错误 —— 使用了错误的版本号而非基于实际文件的正确版本号。

#### 根因分析
- **假设验证失败**：在开始前没有确认 `resources/db/migration/postgresql/` 下已有的版本文件
- 依赖了错误的假设，没有查看实际文件

#### 损失
- 需要重命名迁移文件
- 重新执行迁移流程
- 浪费了排查时间

#### 修正措施
1. **版本号命名前必须验证**：先查看目录下已有的 V{x.x.x} 文件
2. 在 Plan 阶段标注需要确认的关键信息

#### 同场 positive
- 实现前充分探索了代码结构
- 实现前与用户确认了 icon 字段来源
- 完成后运行了编译验证

### 翻车 2：权限过滤逻辑误删

**日期**: 2026-04-09
**复盘文件**: `feedback-retrospect-20260409.md`

#### 事故经过
在移除后端权限过滤逻辑时：
1. 将"不能审核自己的申请"归类为业务逻辑而保留 —— 用户纠正说这也应由前端控制
2. `git mv` 已 stage 了重命名文件，导致代码变更 commit 混入了文件移动

#### 根因分析
- **职责理解不深入**：对项目"前端控制权限"策略理解不够，只覆盖了 admin/non-admin 场景，没覆盖 self-service 限制场景
- **任务混合**：没有在执行 `git mv` 前先提交独立的代码变更

#### 修正措施
1. **前端控制权限策略的完整范围**：当项目采用"前端控制权限"策略时，**所有**用户级别的权限判断都应交给前端，包括"不能操作自己提交的"这类限制
2. **多模式交叉验证**：搜索权限/角色相关逻辑时，同时搜变量名、方法名和中文注释，单一模式容易遗漏
3. **任务分离原则**：代码修改和文件移动分阶段独立提交

#### 深层教训
- 后端 API 权限过滤 和 前端可见性控制数据源（SysMenuServiceImpl）职责不同
- 后者是前端控制的支撑，不能删

### 翻车 3：修改已冻结的 Flyway 脚本

**日期**: 2026-04-10
**复盘文件**: `feedback-retrospect-20260410c.md`

#### 事故经过
在创建 SQL 补丁时，错误地同步修改了已冻结的 `V2.2.13__init.sql`，担心预制模板和补丁数据不一致，多此一举。

#### 根因分析
- **违反不可变原则**：Flyway 已执行/已冻结的脚本严禁修改
- 过度谨慎导致出错

#### 修正措施
1. **Flyway 已执行脚本严禁修改**：所有变更只通过新建补丁文件引入
2. **补丁语句可以独立覆盖所有场景**：无需担心预制模板和补丁数据不一致
3. **操作前自检**：任何涉及 Flyway 的操作，先问自己"补丁文件能否独立完成目标？"如果能，不动历史文件

## 二、早期"蜜月期"的整体教训

**来源**: `my-ideas/02-复盘教训/lessons-learned.md`

### 核心教训：缺少审查导致大量返工

**背景**：刚开始使用 AI 编程时，生成了大量代码，对接时出现大量返工和调试。

**做错了什么**：
- 没有充分验证，直接采纳了大量 AI 生成代码
- 试图一次性做大型模块
- 缺少分阶段验证

**提炼的经验**：
1. **不要妄图一次性做大模块** —— AI 能快速生成大量代码，但不代表能一次性正确
2. **遇到问题再"磨刀"** —— 不要试图提前解决所有可能的问题
3. **小步快跑，及时验证** —— 每完成一个功能点就验证

## 三、错误模式总结

从所有复盘中提炼出的 **四类高频错误模式**：

### 模式 1：假设验证失败

| 维度 | 内容 |
|------|------|
| **症状** | 基于未验证的假设做出决策 |
| **典型案例** | SQL 迁移版本号错误 |
| **根因** | 跳过了"确认现状"这一步 |
| **预防措施** | 在计划阶段识别不确定性，主动确认；涉及版本号/配置时先查看实际文件 |

### 模式 2：职责混淆

| 维度 | 内容 |
|------|------|
| **症状** | 不清楚不同组件的职责边界 |
| **典型案例** | 前端 vs 后端权限控制混淆 |
| **根因** | 对项目架构策略理解不够深入 |
| **预防措施** | 明确区分职责，理解架构策略，多模式交叉验证 |

### 模式 3：不可变违反

| 维度 | 内容 |
|------|------|
| **症状** | 修改已冻结/已执行的不可变内容 |
| **典型案例** | 修改已执行的 Flyway 脚本 |
| **根因** | 过度谨慎/不了解不可变原则 |
| **预防措施** | 操作前自检，遵循增量变更原则，只做加法不做减法 |

### 模式 4：任务混合

| 维度 | 内容 |
|------|------|
| **症状** | 不同类型的变更混合在一起 |
| **典型案例** | 代码修改和文件移动混合提交 |
| **根因** | 没有分离关注点 |
| **预防措施** | 分离关注点，独立提交不同类型的变更 |

## 四、正面实践模式

从复盘中也提炼出了**值得坚持的做法**：

### 探索性实践
- ✅ 实现前充分探索代码结构
- ✅ 多模式交叉搜索验证（变量名 + 方法名 + 中文注释）
- ✅ 主动诊断根因，而非只看表面

### 确认性实践
- ✅ 需求有多种可能时先确认再实现
- ✅ 在 Plan 阶段标注不确定性
- ✅ 涉及版本号/配置时查看实际文件

### 验证性实践
- ✅ 每次代码修改后执行编译验证
- ✅ 用户指出问题后立即回退，不争辩
- ✅ 遵循递增规则和不可变原则

## 五、复盘驱动改进机制

### 复盘三问法

用户创建了自定义 `/retrospect` 技能，通过三问复盘提炼经验：

1. **做对了什么？** —— 强化正确的做法
2. **做错了什么？** —— 识别错误和根因
3. **如果重来会怎么做？** —— 提炼可复用规则

### 经验固化流程

```
完成任务 → 触发复盘 → 三问反思 → 提炼可复用经验 → 保存到 memory/
                                                    → 更新 rules/（如涉及项目规范）
                                                    → 更新 CLAUDE.md（如涉及全局规则）
```

### 记忆类型对应

| 经验类型 | 存储位置 | 生命周期 |
|----------|----------|----------|
| 个人偏好 | ~/.claude/CLAUDE.md | 长期稳定 |
| 项目规范 | .claude/rules/ | 随项目迭代 |
| 复盘教训 | .claude/memory/ (feedback) | 持续积累 |
| 项目背景 | .claude/memory/ (project) | 随项目更新 |
| 外部引用 | .claude/memory/ (reference) | 按需更新 |

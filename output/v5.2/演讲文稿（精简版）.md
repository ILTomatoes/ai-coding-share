## 开场

大家好，今天跟大家聊聊 AI 协作编程。

材料全部来自三个真实的 Java 项目——AI 智能平台后端服务、智能体门户、智能体消息网关。不是理论推导，都是实打实的项目经历。

在座各位大概有三种情况：有些同学已经在用 AI 编程了，有些刚开始，还有些在观望。没关系，今天每个人都能带走一些有用的东西。

## 第一部分：从真实项目说起（20-25 分钟）

先讲两个案例，一正一反，同样的 AI 编程，结局完全不一样。

### 1.1 案例一：aip-server 技能工具模块 — AI 编程的高光时刻

第一个是成功的案例。

背景：我们的 AI 智能平台后端服务 aip-server，需要开发一个技能工具模块，涉及 5 个子模块——代码库管理、MCP 服务、OpenAPI、扩展插件、工作流，总共约 80 个 API 端点。技术栈是 Spring Boot 2.7 + MyBatis Plus + PostgreSQL，Java 17。

协作模式：我先设计原型和表结构，AI 参与讨论评审。表结构定好后，AI 编码，我评审和测试。评审中发现的编码规范问题写到 CLAUDE.md 里，后续 AI 的代码质量就越来越好了。

结果：5 天完成约 80 个 API 端点，29 次提交。值得注意的是 29 次提交里有 13 次 fix 提交，占 45%——AI 的代码确实有不少小错漏，但通过人工评审都被及时捕获了。对比传统方式，效率提升约 2-4 倍。

成功关键：表结构设计清晰，AI 编码边界明确；人定方向，AI 跑量；及时评审、小步验证。还有一点很重要——评审中发现问题后更新 CLAUDE.md 的编码规范，后续 AI 的代码质量确实有明显提升。这个"踩坑→补规范"的模式后面还会反复提到。

### 1.2 案例二：A2A 适配器 — 一场代价惨痛的返工

第二个案例就比较痛了。

背景：需要开发一个 A2A（Agent-to-Agent）协议适配器。A2A 对我来说是个陌生领域，没有深入研究过。

做法：我直接让 AI 根据 A2A 协议文档编码，没有先调研有没有现成方案。AI 确实"高效"——16 天，46 次提交，产出大量代码。但因为我自己对 A2A 不了解，根本没法判断代码对不对。

后来我深入研究 A2A 官方网站才发现，人家有官方的 Java SDK——a2aproject/a2a-java，早就有了。16 天的工作，从零造了一个完全不必要的轮子。

推倒重来用官方 SDK 重写，3 天搞定。代码量差异惊人：自研方案 141,026 行全部删除，SDK 重写只新增 1,786 行，缩减 98.7%。

16 天的工作大部分都是浪费。如果一开始先花 2 小时调研，可能 3 天就搞定了。

另外还有几个 aip-portal 项目的小翻车：

- **SQL 迁移版本号错误**：AI 生成 Flyway 脚本时生成了 V2.2.14，实际应该是 V2.2.13——没先检查已有版本文件。
- **修改已冻结的 Flyway 脚本**：AI 创建补丁 SQL 时担心不一致，改了已冻结的 V2.2.13_init.sql，在已部署环境上直接报错。
- **权限过滤理解偏差**：AI 对"前端可见性控制权限"理解不深，把应该交给前端的逻辑留在后端。解释听起来完全合理，但缺失了业务上下文。

总结：方向错了，效率越高代价越大。

两个案例同样用 AI 编程，一个 5 天搞定 80 个接口，一个 16 天造了个没用的轮子。差别在哪？什么时候该信任 AI？什么时候必须人来把关？后面慢慢展开。

## 第二部分：反思——从案例到洞察（10-15 分钟）

### 2.1 从案例中看到了什么

aip-server 成功的关键：人在前面给了清晰的设计和边界，AI 在明确框架里填代码。A2A 失败的根因：人没有给方向，对领域不了解就让 AI 直接干。

核心观察：**AI 的效率是确定的，不确定的是人的引导。**同一个 AI，同一个开发者，不同项目结果天壤之别。区别不在于 AI 的能力，而在于人怎么用它。

AI 的潜力：编码效率 2-4 倍提升是真实的，特别擅长重复性和模式化代码。

AI 的边界：不了解领域背景时容易跑偏；缺乏全局视角，可能"局部最优但全局灾难"；最重要的是不会主动质疑你的设计。

一句话：**AI 是最强执行者，但不是合格的架构师。**

### 2.2 核心洞察：把最擅长的事交给 AI

这个观点可能跟直觉反差比较大。

很多人觉得应该先把不擅长的事交给 AI。实际上恰恰相反——你应该先把最擅长的事交给 AI。三个原因：

第一，你擅长的事情你能准确验收。写了几年 CRUD 接口，AI 写出来好不好一眼就知道。

第二，已经擅长的工作对你没有成长价值了，就是纯工作量堆叠。

第三，释放出来时间可以投入研究、设计、架构这些更有创造性的工作。

角色转变：过去自己设计、编码、测试；现在自己设计、AI 编码、自己评审测试。从"码农"变成"系统架构师"——你画图纸，AI 盖楼。

### 2.3 但这里有个陷阱：AI 会 PUA 你

AI 有一个特点：永远自信满满。

方向错了也语气笃定、逻辑自洽，很容易被"说服"。它不像人类同事遇到不确定的事会犹豫。A2A 案例中 AI 全程没有任何犹豫，自信地造了完整轮子。如果它说一句"可能有现成的 SDK"，16 天就不需要浪费了。

应对方式：保持自己的判断力，对关键逻辑一定要人工验证。

### 2.4 关键问题：怎么让 AI 稳定输出高质量代码？

把前面的思考串起来：应该把擅长的事交给 AI，但 AI 会 PUA 你，靠个人判断力够不够？

不够。aip-server 成功靠的是个人经验和设计到位，A2A 翻车靠事后补救。单靠个人经验和判断力不可持续、不可复制。

我们需要一套系统化的方法——不是靠"手感"，而是可复制、可团队化的流程和工具。这就是接下来要讲的 SDD 和 Harness。

## 第三部分：系统化提升 — SDD 与 Harness 实践（30-35 分钟）

这是今天最核心的内容。两个概念配两个项目案例，先讲 SDD 再讲 Harness。

### 3.1 什么是 SDD

SDD 全称 Specification-Driven Development，规范驱动开发。核心思想：写代码之前，先用自然语言把"做什么"和"为什么"定义清楚。这份"说明书"叫 Spec，是人与 AI 之间共同的唯一真理来源。

为什么需要 SDD？为了终结"氛围编程"——凭感觉提需求，AI 猜着写，写错花几个小时修。A2A 案例就是典型的氛围编程。

Spec 就像建筑的施工图纸——规定了每一层的结构、每一根钢筋的位置，AI 按图纸施工。它不是写完就压箱底的需求文档，而是一份"活"的契约，代码只是 Spec 的"派生物"，Spec 才是核心。

高质量 Spec 包含四个部分：目标与价值、上下文与约束、功能逻辑、验证标准。注意最后一条——因为 Spec 强制包含测试标准，AI 生成的代码就自带 TDD 基因。

SDD 标准工作流四阶段：定义（Specify）→ 计划（Plan）→ 任务（Task）→ 实现（Implement）。

三个核心收益：

- **可预测性**：AI 动手前你已经知道它要改什么、怎么改，不再是开盲盒。
- **高质量代码**：Spec 强制包含测试标准，大幅减少安全漏洞和业务逻辑错误。
- **易于维护**：一年后想了解某逻辑为什么存在，答案在 Spec 里。

什么时候写 Spec？改一个字段这种小事直接让 AI 做；半天以内的功能写个 5-10 行迷你 Spec；超过一天的复杂功能必须走完整流程。判断标准：这个需求给新同事讲超过 5 分钟，就值得写 Spec。还有一个隐性收益——Spec 本身就是最好的技术文档。

### 3.2 用智能体消息网关讲 SDD 实践

案例来自智能体消息网关 aip-gateway。AMG 核心模块用全响应式技术栈（Spring WebFlux + R2DBC + Redis Reactive），有意识地从第一天就系统化来做。

SDD 四阶段落地：PRD → Spec → 审阅 → 编码。

- **PRD 阶段**：人定义需求边界。
- **Spec 阶段**：AI 辅助把 PRD 转化为技术方案，产出 4796 行的详细设计文档。
- **审阅阶段**（最关键）：用 Claude Code 多轮审阅，设计文档从 v1 迭代到 v1.9，经历 9 轮。三轮审阅发现的关键问题：
  - **架构定位偏移**：AMG 被设计成独立服务，审阅中发现应该是消息层。如果到编码阶段才发现，代价巨大。
  - **技术栈兼容性**：Spec 误用了 Spring Boot 3.x API，实际是 2.7.17。
  - **数据模型关系**：三层关系表述不清，审阅中做了明确。

审阅不是走形式，每一次都在防止"AI 高效跑偏"。

- **编码阶段**：有了完善的 Spec，Agent Team 根据 Spec 自动编码。

成果：AMG 核心模块 59 个 Java 文件，2 天完成。新增 13,093 行代码，71 个文件变更。从第一天就没有混乱期。

回扣 aip-server 案例：那是靠个人经验和设计到位成功的；aip-gateway 是靠 SDD 流程成功的。同样的好结果，但后者可复制、可团队化。

### 3.3 什么是 Harness Engineering

LangChain 有个公式：**Agent = Model + Harness**。模型是马，Harness 是缰绳。光有马不行，还得有一整套驾驭系统。

Harness 有两大控制机制，形成闭环：

- **Guides（前馈控制）**：AI 行动前设好规则。像高速公路护栏——不需要每秒纠正，护栏在那里就行。CLAUDE.md、代码规范文档、架构决策记录都属于 Guides。
- **Sensors（反馈控制）**：AI 做完后检测对不对。自动化测试、代码 lint、CI 流水线就是 Sensors。

核心原则：**每发现 Agent 犯一个错误，就设计方案让它永远不可能再犯。**这就是 Harness 闭环。

还有一个原则：**约束先行**。规则靠自动化强制执行，不是靠口头约定。

Harness 的上下文供给分三层：项目级规范（CLAUDE.md + rules，相当于员工手册，自动加载）、任务级规范（Plan、PRD、Spec）、会话级引导（对话中实时补充）。

### 3.4 用智能体门户项目讲 Harness 的搭建与生长

案例来自 aip-portal，259 次提交、72 天开发周期。这个项目经历了 Harness 从无到有的完整演化——**从混乱中提炼秩序**。

核心是 rules 目录的四阶段演化：

**Phase 1 初始阶段**：5487 行、20 个文件，什么都往里放。

**Phase 2 膨胀阶段**：24 个文件、约 264KB，最大单文件 53KB。踩坑就加规则，越加越多，形成恶性循环。

**Phase 2.5 发现根因**：code-style 文件配置了 paths 匹配所有 Java 文件，每次编辑都加载全部内容——Token 巨大浪费。

**Phase 3 大清理**：删掉 2714 行。确立核心原则：**rules 目录是 Token 预算管理，不是 wiki。**

**Phase 4 精炼阶段**：最终 272 行，压缩率 95%。只保留 AI 必须知道的约束。

最终形态是三层知识架构：CLAUDE.md（23 行，项目导航）→ rules（272 行，3 个文件，核心约束自动加载）→ docs/code-style（838 行，8 个文件，按需加载）。

核心经验：
- rules 是 Token 预算管理，每一行都要有存在的价值。
- 规范越精炼越好，少即是多。
- 踩坑是完善规范的机会，但加完后要定期精简。
- 带 paths 配置的参考文档不要放 rules 目录，移到 docs 只保留索引。

对比两个项目：
- **aip-gateway**：SDD 驱动，"规范先行，代码自动"，从第一天建 Spec，基本没有混乱期，2 天完成核心模块。
- **aip-portal**：Harness 演化，"从混乱中提炼秩序"，约 2 个月混乱期。

启示：有了方法论后新项目可以少走很多弯路，"PRD → Spec → 审阅 → 编码"这个工作流可直接套用。

## 第四部分：工具链与实战技巧（10-15 分钟）

### 4.1 工具介绍

主要用 **Claude Code**——终端工具，直接和项目代码零距离。核心优势：读取文件、编辑代码、执行命令全流程闭环，自动加载 CLAUDE.md 和 rules 理解项目上下文。

搭配 **cc-switch** 快速切换配置——不同项目不同模型或权限，一键切换。

### 4.2 实战技巧

**技巧一：小步快跑，持续验证。**

AI 生成快，错误积累也快。设计→实现→测试，每完成一个模块就验证。遇到问题再"磨刀"，不要试图提前设计完美体系。

**技巧二：先建立认知，再让 AI 干活。**

从 A2A 教训中总结：越不了解的领域越要先调研。你的认知边界就是 AI 效率的上限。新领域先让 AI 帮你建立认知框架，再进入编码。

**技巧三：根据场景选协作模式。**

- 明确小任务 → 直接对话
- 复杂功能 → Plan 模式，先规划再执行
- 多任务并行 → Worktree + Agent Team
- 新领域探索 → 先调研
- 代码评审 → 只读模式

### 4.3 快速行动建议

**刚开始实践的同学**：从小项目或小模块开始；先写好 CLAUDE.md（技术栈、项目结构、核心约定）；养成"设计→AI 实现→验证"的习惯。

**有经验的同学**：搭建项目级 Harness——20-30 行 CLAUDE.md 做导航，rules 放核心约束索引，详细规范放 docs 按需引用；试试 Worktree 并行开发；建立踩坑补规范的复盘习惯。

**还在观望的同学**：不需要全面切换，先在小任务上试试——让 AI 写单元测试、做代码评审、写 CRUD 接口。感受到效率提升后自然会想扩大范围。

## Q&A（5 分钟）

# 《Java开发中的 AI 协作编程实战》分享规划

> 预计时长 1.5 小时 | 受众：部门 Java 开发同事

---

## 一、演讲大纲设计

### 开场（10 分钟）：从一个真实场景切入

**不要从"AI 是什么"讲起，直接上冲击力。**

建议用一个真实的对比场景：
- 展示你某次用 Claude Code 完成一个完整功能的 git log（从设计到实现到测试）
- 对比传统开发方式下同类任务的工作量
- 抛出核心观点：**AI 编程不是让 AI 替你写代码，而是你和 AI 组成一个"双人编程"团队**

### 第一部分（20 分钟）：驯服 AI——从混乱到有序

> 核心叙事线：我踩过的坑，怎么一步步建立起有效的协作体系

#### 1.1 初期的"蜜月期"与"翻车期"
- 刚开始的兴奋：AI 能快速生成大量代码
- 紧接着的教训：缺少审查、大量返工（引用 `02-复盘教训/lessons-learned.md`）
- **关键转折点**：意识到 AI 编程的核心不是"生成"，而是"驾驭"

#### 1.2 知识体系的搭建：三层架构
- 全局层：个人偏好和跨项目规则
- 项目层：CLAUDE.md + rules/（编码规范、架构约束）
- 会话层：memory/（复盘经验、项目上下文）
- **现场演示**：展示你的 `.claude/` 目录结构

#### 1.3 rules/ 的演变故事（这是最有说服力的部分）
- Phase 1（初始搭建）：粗糙的单文件
- Phase 2（功能驱动膨胀）：819 行大杂烩 + 1706 行设计文档，token 爆炸
- Phase 3（关键转折——大清理）：**一次删掉 6278 行**，确立核心原则
- Phase 4（稳定维护）：结构清晰、按需更新
- **核心经验**：`rules/ 是 token 预算管理，不是 wiki`

### 第二部分（20 分钟）：实战工作流——设计、实现、测试闭环

#### 2.1 设计阶段：PRD 质量决定成败
- PRD 清晰 → AI 事半功倍
- PRD 模糊 → AI 飞速跑偏，无法验收
- 实践：用 IDE 写设计 Markdown → Claude Code 输出实施计划 → 人工审阅

#### 2.2 实现阶段：多种协作模式
- **Plan 模式**：先探索 → 确认 → 再实现
- **Sub Agent**：拆分子任务，控制上下文
- **Agent Team**：复杂任务的多 Agent 并行
- **Git Worktree**：多任务物理隔离，并行开发
- **现场演示**：展示 worktree 并行开发的截图

#### 2.3 测试阶段
- 单元测试：AI 生成 + 人工审查（注意维护成本）
- Karate 测试：业务接口场景的利器
- 教训：测试成功率低时的改善思路

#### 2.4 复盘驱动改进
- `/retrospect` 复盘命令：做对了什么、做错了什么、如果重来会怎么做
- 从错误中提炼规则，写入 rules/ 或 memory/
- **每次踩坑都是完善知识体系的机会**

### 第三部分（15 分钟）：进阶技巧与工具链

#### 3.1 工具组合
- Claude Code + cc-switch 工具组合
- Playwright-MCP：编写用户手册、系统调研
- 不同模型的分工：Opus 做设计、其他模型做编码

#### 3.2 "奢侈但有效"的策略
- 反复尝试无果时，切换到更强的模型（Opus）
- 不要在低效的循环里死磕

#### 3.3 AI 的其他应用场景
- 解读开源项目并输出文档
- AI + Playwright 深入解读系统
- AI 自学新技能/技术

### 收尾（10 分钟）：给同事的实操建议

#### 起步三步走
1. **第一天**：创建 CLAUDE.md，写三行（项目是什么、用什么技术、怎样算好）
2. **第一周**：遇到问题就加一条 rule，不要提前设计
3. **第一个月**：回顾 rules/，做一次大清理，形成稳定结构

#### 心态转变
- 从"代码作者"→"系统设计师"
- 从"追求一步到位"→"先有再优化"
- 从"自己记住经验"→"让项目记住经验"

### 互动 Q&A（15 分钟）

## 二、素材提取方案

### 2.1 从 `.claude/` 目录提取

| 目标 | 提取方式 | 产出 |
|------|----------|------|
| rules/ 的演变过程 | `git log --oneline --stat -- .claude/rules/` | 时间线图：从膨胀到精简 |
| 关键决策点 | `git log --all -p -- .claude/rules/` 查看大删改的 diff | 截图：一次删 6278 行的 commit |
| CLAUDE.md 变化 | `git log -p -- CLAUDE.md` | 对比：初版 vs 当前版本 |
| memory 记录 | 直接读取 `.claude/memory/` 下的文件 | 复盘条目清单 |
| settings 配置 | 读取 `.claude/settings.json` | Hooks 配置示例 |

**建议执行的 git 命令（在你的 aip-portal 项目中）：**

```bash
# 1. 查看 .claude 目录的完整提交历史
git log --oneline --stat -- ".claude/"

# 2. 查看 rules 目录的行数变化趋势
git log --oneline --numstat -- ".claude/rules/"

# 3. 找到"大清理"那次提交的详情
git log --all --oneline --diff-filter=D -- ".claude/rules/"

# 4. 查看 CLAUDE.md 的演变
git log --oneline -p -- "CLAUDE.md"

# 5. 统计当前 rules 目录的规模
find .claude/rules -name "*.md" | xargs wc -l

# 6. 查看 memory 目录的内容
ls -la .claude/memory/
cat .claude/memory/MEMORY.md
```

### 2.2 从 Git 提交记录提取

| 目标 | 提取方式 | 产出 |
|------|----------|------|
| "fix fix fix" 模式 | `git log --oneline` 找早期连续 fix 的提交 | 截图：展示早期混乱的提交记录 |
| Plan 模式后的改善 | 对比后期结构化的提交记录 | 截图：展示有序的提交记录 |
| AI 参与的提交量 | 统计 Co-Authored-By 或特定模式的提交 | 饼图：AI 参与度 |
| 功能开发速度 | 按功能模块统计从首次提交到完成的时间 | 对比数据 |

**建议执行的 git 命令：**

```bash
# 1. 查看最近的提交记录（观察提交消息质量的变化）
git log --oneline -50

# 2. 按天统计提交频率
git log --format='%ad' --date=short | sort | uniq -c

# 3. 统计每周代码变更量
git log --since="2026-03-25" --numstat --format='' | awk '{add+=$1; del+=$2} END {print "Added:", add, "Deleted:", del}'

# 4. 查找连续 fix 的提交（早期混乱期）
git log --oneline --grep="fix" --since="2026-03-25" --until="2026-03-31"

# 5. 查看某个功能模块的完整开发历史
git log --oneline -- "src/main/java/path/to/module/"
```

## 三、其他建议

### 3.1 演讲风格建议

- **多讲故事，少讲理论**：同事最想听的是"你真实用下来怎么样"，不是"AI 编程理论上应该怎么做"
- **多展示真实截图**：git log、.claude 目录结构、rules 文件内容、worktree 并行截图
- **敢于展示失败**：展示早期的混乱和错误，比展示成功更有说服力
- **给出可落地的行动建议**：每个部分结束时给出"你明天就可以做的一件事"

### 3.2 可以考虑的互动环节

- **现场演示**：用一个简单的 Spring Boot 项目，现场展示从创建 CLAUDE.md 到完成一个小功能的全过程（5-8 分钟）
- **投票/提问**：开场时问大家"你用过 AI 编程工具吗？用到什么程度？"了解受众水平
- **实操体验**：如果条件允许，分享后安排一个"动手试试"环节

### 3.3 需要提前准备的材料

| 材料 | 来源 | 优先级 |
|------|------|--------|
| aip-portal 的 `.claude/` 目录截图 | 本地截图 | 高 |
| rules/ 演变的 git diff 截图 | git log 提取 | 高 |
| "大清理"前后的对比图 | git diff 提取 | 高 |
| worktree 并行开发截图 | 已有 `images/worktree/` | 中 |
| 提交记录对比截图（混乱期 vs 有序期） | git log 提取 | 中 |
| 简单的现场 demo 项目 | reference/examples/ 或新建 | 低 |

### 3.4 时间分配风险提示

- 第一部分（驯服 AI）内容最丰富，容易超时，建议控制在 20 分钟内
- 现场演示如果出问题，准备好截图/录屏作为 Plan B
- Q&A 环节预留充足时间，同事的实际问题往往是最有价值的部分

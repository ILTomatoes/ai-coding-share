## 一、演讲大纲

### 开场（8 分钟）：AI 编程的现实——不是银弹，但已是利器

**核心目标**：用真实数据建立可信度，抛出核心观点

**内容要点**：
1. **一组真实数据冲击**：
   - 793+ 个 Claude Code 会话、50+ 个项目、日均 1000-6000 条消息
   - aip-portal 项目：24 个 worktree 并行开发，256 次提交
   - A2A 适配器案例：17 天返工 vs 如果先调研只需 3 天
2. **抛出核心观点**：
   - AI 编程不是"让 AI 替你写代码"，而是"人掌舵，AI 执行"
   - 从"氛围编程（Vibe Coding）"到"驾驭工程（Harness Engineering）"
   - 工程师的角色转变：从"代码作者"到"系统设计师"

#### 1.2 Harness Engineering：给 AI 套上马具（5 分钟）

**内容**：
- **马具比喻**：AI 模型 = 烈马，Harness = 马具（缰绳、鞍具），人 = 骑手
- **Harness 工程的三大支柱**：
  1. 上下文工程（Context Engineering）：确保 AI 看到正确的信息
  2. 架构约束（Architectural Constraints）：用工具机械地强制执行
  3. 熵管理（Entropy Management）：控制代码库的混乱度
- **行业佐证**：
  - OpenAI Codex：3 名工程师 5 个月构建 100 万行代码，零手写
  - LangChain：不改模型只改 Harness，排名从 Top30 跳到 Top5

#### 1.3 三层知识体系：我的实践架构（10 分钟）

**内容**：
- **三层架构图**：
  - 全局层（`~/.claude/CLAUDE.md`）：跨项目个人偏好（Git 规范、Shell 规范）
  - 项目层（`CLAUDE.md` + `.claude/rules/`）：架构、规范、关键约束，每次对话全部加载
  - 会话层（`memory/`）：复盘经验、项目上下文，按需检索
- **rules/ 演变的完整故事**（最有说服力的部分）：
  - Phase 1（初始）：20 个文件，5487 行 → 想把所有知识给 AI
  - Phase 2（膨胀）：功能驱动持续增长，巅峰 ~6000+ 行，token 爆炸
  - Phase 3（大清理）：一次删 2714 行，确立"rules 不是 wiki"
  - Phase 4（持续精简）：最终 272 行（压缩率 95%）
- **核心经验**：
  - `rules/ 是 token 预算管理，不是 wiki`
  - 人看的放 `docs/`，AI 必须看的放 `rules/`
  - `paths` frontmatter 的陷阱：`**.java` 导致所有规范文件全部加载

### 第三部分（15 分钟）：深度案例——A2A 适配器的教训

> 用一个完整案例串联所有经验，这是最有说服力的部分

#### 3.1 案例全貌（15 分钟）

**内容**：
- **背景**：为 ShenYu 网关开发 A2A 协议适配器
- **时间线**：
  - 阶段一（Day 1）：自信开干，一天写了 3840 行 → 编译能过但协议不对
  - 阶段二（7天）：不断修补 → 代码膨胀到 7600 行 → 越补越偏
  - 阶段三（转折）：发现官方有 a2a-java SDK → 用 9 个会话学习源码
  - 阶段四（1周）：基于 SDK 重写 → 2400 行搞定 → 代码量减少 68%
- **返工代价量化**：
  | 维度 | 旧方案（手写） | 新方案（SDK） |
  |------|-------------|-------------|
  | 代码量 | ~7,600 行 | ~2,400 行 |
  | 开发时间 | 9天（含返工） | 9天（含学习） |
  | 协议正确性 | 多处不一致 | SDK 保证 |
  | 维护成本 | 高 | 低 |
- **三个关键教训**：
  1. **先调研再动手**：官方有 SDK 吗？有示例吗？社区有方案吗？
  2. **AI 不会替你做技术选型**：AI 不会主动告诉你"有官方 SDK 可以用"
  3. **越不了解的领域，AI 输出越危险**：看起来越合理，就越要警惕

### 收尾（10 分钟）：给同事的实操建议

#### 起步三步走

1. **第一天**：创建 CLAUDE.md，写三行——项目是什么、用什么技术、怎样算好
2. **第一周**：遇到问题就加一条 rule，不要提前设计
3. **第一个月**：回顾 rules/，做一次大清理，形成稳定结构

#### Harness 是"长"出来的

- 不要试图一步到位设计完美的规则体系
- 每一条规则都应有一个具体的"AI 犯错事件"作为来源
- Harness 的生长循环：让 AI 做 → 观察问题 → 分析根因 → 添加最小约束 → 验证效果

#### 心态转变

| 从 | 到 |
|----|-----|
| 代码作者 | 系统设计师 |
| 追求一步到位 | 先有再优化 |
| 自己记住经验 | 让项目记住经验 |
| 文档越全越好 | 信息密度越高越好 |

#### 避坑指南

- 警惕"Markdown 怪兽"：规范要短小、原子化、可验证
- 不要过度微观管理：明确意图和约束后，给 AI 发挥空间
- 两轮失败原则：修正超过两次还没过，果断 `/clear` 重来
- 越不了解的领域，越要先建立认知再让 AI 辅助

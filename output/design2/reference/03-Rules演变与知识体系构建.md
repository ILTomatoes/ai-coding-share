### 关键区别

| 层级 | 加载时机 | 消耗 token | 更新频率 | 适合放什么 |
|------|----------|------------|----------|-----------|
| 全局层 | 每次对话 | 少 | 极低 | 个人偏好、跨项目规范 |
| 项目层 | 每次对话 | 多 | 随项目迭代 | 架构、规范、关键约束 |
| 会话层 | 按需检索 | 极少 | 持续积累 | 复盘经验、项目历史 |

## 四、核心经验提炼

### 4.1 rules/ 的本质是什么？

> **rules/ 是 token 预算管理，不是 wiki**

| 误区 | 正确理解 |
|------|----------|
| rules/ 是项目文档库 | rules/ 是 AI 每次对话的必读清单 |
| 内容越多越好 | 每一行都在消耗 token，必须有存在价值 |
| 一次性设计完美 | rules 是"长"出来的，不是一次性设计的 |
| 给人看和给 AI 看的一样 | 人看的放 docs/，AI 必须看的放 rules/ |

### 4.2 什么该放 rules/，什么不该放？

**应该放**:
- ✅ 架构约束（模块划分、依赖方向）
- ✅ 关键设计决策（为什么这样做，不解释怎样做）
- ✅ 编码规范的核心原则（详细规范放 docs/，索引放 rules/）
- ✅ 测试策略和执行方式

**不应该放**:
- ❌ 可从代码推导的信息（技术栈版本、文件路径、包结构）
- ❌ 运维和部署指南（AI 不负责部署）
- ❌ 设计文档和执行计划（人看的文档，放 docs/）
- ❌ 环境信息（数据库连接、密码、IP 地址，放 application.yml）
- ❌ 临时状态（进度跟踪、任务管理，用 Git issues/tasks）

### 4.3 paths frontmatter 的陷阱

**问题**: code-style/ 下的文件配了 `paths: **.java`
```
---
paths: **.java
---
```

**后果**: 编辑**任何一个** Java 文件时，**所有** code-style/ 文件全部注入上下文

**解决**: 详细规范移到 `docs/code-style/`，rules/ 中只保留索引文件（不带 paths）

### 4.4 文件生命周期管理

```
创建（设计阶段）
  → 迭代（开发中持续更新）
    → 稳定（完成后冻结）
      → 过时（下次重构时更新或删除）
```

**管理规则**:
| 规则 | 说明 |
|------|------|
| 一份文件一个主题 | 避免大杂烩文件，拆分后 AI 检索更精准 |
| 及时清理 | 过时的、完成的、空壳的文件及时删除 |
| 定期回顾 | 每隔一段时间做一次大清理 |

## 五、给同事的建议

### 起步三步走

**第一天**：创建 CLAUDE.md，写三行
```markdown
# 项目是什么
# 用什么技术栈
# 怎样算好的代码
```

**第一周**：遇到问题就加一条 rule，不要提前设计
- 不要试图一开始就设计完美的 rules 体系
- 遇到一个问题，就加一条对应的规则

**第一个月**：回顾 rules/，做一次大清理
- 删除可以从代码推导的信息
- 移除"人看"的文档
- 确认每条规则都有存在的价值

### 心态转变

| 从 | 到 |
|----|-----|
| 代码作者 | 系统设计师 |
| 追求一步到位 | 先有再优化 |
| 自己记住经验 | 让项目记住经验 |
| 文档越全越好 | 信息密度越高越好 |

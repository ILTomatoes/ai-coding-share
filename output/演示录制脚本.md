## 视频 1：没有 CLAUDE.md 时的 AI 输出

**对应 PPT 24，时长约 2 分钟**

### 录制步骤

1. 打开 taskflow-demo 项目目录
2. 确认没有 CLAUDE.md 文件（`ls -la` 确认）
3. 启动 Claude Code（终端输入 `claude`）
4. 输入以下提示词：

```
实现创建任务的 API：POST /api/tasks，接受 JSON body 包含 title（必填）和 description（选填）字段，保存到数据库并返回创建的任务。同时实现查询所有任务的 GET /api/tasks 接口。
```

5. 等待 AI 生成代码（约 30-60 秒）
6. 观察并展示 AI 生成的代码

### 预期 AI 输出（典型问题）

AI 大概率会产生以下问题（如果不产生，可以追加请求引导）：

- **问题 1**：TaskController 中直接注入 TaskRepository，跳过 Service 层
- **问题 2**：直接返回 JPA Entity 作为 API 响应，没有 DTO/VO
- **问题 3**：没有统一异常处理，可能出现 RuntimeException
- **问题 4**：Controller 中包含参数校验逻辑

### 录制要点

- 打开生成的 Controller 文件，高亮展示：`TaskRepository` 直接注入、业务逻辑在 Controller 中
- 打开 Entity 文件，展示直接被当作 API 响应返回
- 不需要展示所有文件，重点展示 2-3 个典型问题即可
- 录制完成后，不要执行 AI 的代码修改（保持项目状态，用于视频 2）

### 讲解词（配合播放时使用）

> "这是一个完全'裸奔'的项目——没有 CLAUDE.md，没有任何配置。大家看 AI 给我们的代码：业务逻辑直接写在 Controller 里，直接注入 Repository，没有 Service 层。Entity 直接作为 API 响应返回，数据库字段全部暴露。没有 DTO，没有异常处理。代码能跑，但完全不是我们期望的架构。"

## 视频 2：3 行 CLAUDE.md 的效果

**对应 PPT 24，时长约 2 分钟**

### 录制步骤

1. 先回退视频 1 中 AI 的修改：`git checkout .` 或手动删除
2. 创建 CLAUDE.md 文件，内容只有 3 行：

```markdown
# TaskFlow Demo

Spring Boot REST API 任务管理服务。使用 JPA + H2/MySQL 存储。
Java 17+，代码风格遵循 Google Java Style，测试用 JUnit 5 + MockMvc。
```

3. 展示 CLAUDE.md 内容（`cat CLAUDE.md`）
4. 重新启动 Claude Code
5. 输入与视频 1 完全相同的提示词：

```
实现创建任务的 API：POST /api/tasks，接受 JSON body 包含 title（必填）和 description（选填）字段，保存到数据库并返回创建的任务。同时实现查询所有任务的 GET /api/tasks 接口。
```

6. 等待 AI 生成代码
7. 对比视频 1 的输出

### 预期 AI 输出（改善）

有了技术栈信息后，AI 应该：
- 自动生成 Controller、Service、Repository 的分层结构
- 使用 Google Java Style 的命名规范
- 可能仍然缺少 DTO 分离（因为 3 行信息没有明确要求）

### 录制要点

- 画面中同时展示：CLAUDE.md 内容 + AI 生成的代码
- 重点关注 Controller 文件，展示它现在调用了 Service 层而不是直接操作 Repository
- 与视频 1 的输出形成对比
- 可以用 split screen 效果（后期编辑或双显示器）

### 讲解词

> "现在我只加了三行 CLAUDE.md：项目名、技术栈、编码风格。同样的请求，大家看——至少基本的分层结构出来了。Controller 不再直接操作数据库，而是通过 Service 层。Google Java Style 的命名也自动用上了。但注意，它仍然直接返回了 Entity，没有 DTO 分离——因为我们的 3 行信息里没有要求这个。"

## 视频 3：逐步添加约束的生长过程

**对应 PPT 24，时长约 4 分钟**

### 录制步骤

这个视频可以拆分成 3 个连续片段录制，后期合并。

#### 片段 3a：添加分层架构规则（~1 分钟）

1. 在 CLAUDE.md 末尾追加：

```markdown

## 分层架构
- Controller：只做参数接收、调用 Service、返回响应，不含业务逻辑
- Service：所有业务逻辑在此，通过接口定义（如 TaskService 接口 + TaskServiceImpl）
- Repository：继承 JpaRepository，只做数据访问
- Entity：JPA 实体类，映射数据库表
```

2. 让 AI 重构现有代码，遵循分层架构
3. 展示重构后的代码结构

#### 片段 3b：添加 DTO 分离规则（~1 分钟）

1. 在 CLAUDE.md 末尾追加：

```markdown

## 数据传输
- API 请求用 XxxRequest DTO（如 CreateTaskRequest）
- API 响应用 XxxResponse VO（如 TaskResponse）
- Entity 不直接暴露给 Controller 层
- DTO/VO 放在 dto 包下
```

2. 让 AI 引入 DTO 层，重构 Controller 和 Service
3. 展示现在 Controller 接受 CreateTaskRequest、返回 TaskResponse

#### 片段 3c：添加统一异常处理规则（~1 分钟）

1. 在 CLAUDE.md 末尾追加：

```markdown

## 异常处理
- 业务异常继承自定义基类 BusinessException（含 errorCode + message）
- Controller 层不 try-catch，由全局 @RestControllerAdvice 统一处理
- 错误响应格式：{"code": "ERROR_CODE", "message": "描述", "timestamp": "..."}
```

2. 让 AI 添加全局异常处理器
3. 展示最终效果：异常统一处理、错误响应格式一致

### 录制要点

- 每个片段开始时展示 CLAUDE.md 的当前内容（标注行数变化：3→9→15→22）
- 展示 AI 的代码修改过程
- 片段之间用 CLAUDE.md 行数增长的画面过渡
- 最终展示完整的 CLAUDE.md（约 25-30 行）和对应的项目代码结构

### 讲解词

> "接下来演示 Harness 的'生长'过程。每次 AI 犯了错，我们就加一条规则。
>
> 第一次：AI 把业务逻辑写在 Controller 里。我加了分层架构规则，从此 AI 自动遵循三层架构。
>
> 第二次：AI 直接返回 Entity 对象。我加了 DTO 分离规则，从此每个 API 都有 Request 和 Response 类。
>
> 第三次：异常处理风格不一致——有的 throw RuntimeException，有的直接返回 ResponseEntity。我加了统一异常处理规则，从此全局用 @RestControllerAdvice 处理。
>
> 看，CLAUDE.md 从 3 行长到了 30 行左右。每一条规则的来源都是一个具体的'事故'。"

## 视频 4：worktree 多任务并行

**对应 PPT 25，时长约 3 分钟**

### 录制步骤

1. 展示项目目录结构
2. 在 Claude Code 中创建 worktree：
   ```
   /worktree feature-add-user
   ```
3. 展示新的工作目录已创建
4. 打开第二个终端窗口
5. 在新终端启动另一个 Claude Code 实例
6. 展示两个 Claude Code 实例同时工作

### 已有截图素材

如果录制不顺利，可以直接使用已有的截图：
- `my-ideas/images/多个worktree并行.png`
- `my-ideas/images/多个worktree并行2.png`
- `my-ideas/images/git-worktree.png`

### 录制要点

- 重点展示两个终端窗口并排的效果
- 每个窗口有不同的 AI 在工作
- 展示 `git worktree list` 输出，确认两个 worktree 同时存在

### 讲解词

> "在实际开发中，你可能同时有多个需求在推进。worktree 的做法是：每个任务一个独立的工作目录，各自有独立的 AI 会话。大家看：左边在做一个用户管理功能，右边在做一个数据导出功能，两个 AI 实例同时工作，互不影响。这比频繁切换分支、丢失 AI 上下文要高效得多。"

## 视频 5：设计-开发-测试闭环

**对应 PPT 25，时长约 3 分钟**

### 录制步骤

1. 在 Claude Code 中使用 Plan 模式：

```
/plan 实现一个任务导出功能：GET /api/tasks/export，支持按日期范围筛选，导出为 CSV 格式
```

2. 展示 AI 输出的设计方案（Plan 模式下 AI 只做规划不写代码）
3. 确认方案后，让 AI 执行实现
4. 展示 AI 编码过程
5. 展示测试验证（如有）

### 录制要点

- 重点展示三个阶段的转换：Plan → 实现 → 验证
- 每个阶段有明确的产出物
- 可以适当剪辑加速编码过程

### 讲解词

> "这是一个完整的闭环流程。第一步，用 Plan 模式让 AI 先规划——探索代码、分析影响、设计实现方案。第二步，确认方案后让 AI 编码实现。第三步，运行编译和测试验证。每个环节有明确的检查标准，避免'一口气做完发现方向错了'。"

## 视频 6：/retrospect 复盘与记忆积累

**对应 PPT 26，时长约 3 分钟**

### 录制步骤

1. 在一个已完成的小任务后，输入 `/retrospect` 或直接说"复盘"
2. 展示 AI 输出的复盘结果（三问反思）
3. 展示 AI 过滤后的可保存经验
4. 展示记忆文件保存到 memory/feedback/ 目录
5. 开启新会话，做一个类似任务
6. 展示 AI 自动检索到相关记忆并应用

### 预期 AI 输出

复盘结果应包含：
- **做对了什么**：比如"提前确认了数据库字段命名规范"
- **做错了什么**：比如"没有检查已有的工具类，重复造轮子"
- **如果重新做**：比如"先搜索项目中是否有类似的工具类"

保存的记忆文件应包含：
- 规则描述
- Why（为什么）
- How to apply（何时用）

### 录制要点

- 展示完整的复盘流程，从触发到保存
- 重点展示记忆文件的格式（规则 + Why + How to apply）
- 最后展示记忆在新会话中被复用的效果——这是"闭环"的证明

### 讲解词

> "最后看复盘机制。完成任务后，我触发复盘。AI 回顾整个过程，给出三问反思。然后它过滤出值得保存的经验，按照'规则 + Why + How to apply'的格式写入 memory 目录。
>
> 下次开新会话，做一个类似任务时，AI 自动检索到了之前的记忆，提醒我注意某个坑。这就是'经验制度化'——把'下次注意'从一句空话变成可执行的知识记录。"

# AI 协作编程实践 — 演讲详细大纲

> 时长：90分钟 | 听众：Java 开发人员（30% 有经验 / 60% 入门中 / 10% 观望）

---

## 第一部分：从真实项目说起（20-25分钟）

> 目标：用两个反差鲜明的案例，让听众感受到 AI 编程的巨大潜力，同时也看到盲目信任的风险

### 1.1 案例一：aip-server 技能工具模块 — AI 编程的"高光时刻"（10-12分钟）

**项目背景**
- AI 智能平台后端服务（aip-server）的技能工具模块（logic/tool）
- 需要开发约 80 个 API 端点，覆盖 5 个子模块（代码库、MCP、OpenAPI、插件、工作流）
- 技术栈：Spring Boot 2.7.18 + MyBatis Plus 3.4.2 + PostgreSQL + Java 17

**协作模式**
- 人设计原型 → 人设计表结构（AI 参与讨论、评审）→ AI 编码 → 人评审 & 人测试
- 评审发现问题后，人在 CLAUDE.md 中补充编码规范，后续 AI 代码质量提升

**关键数据和成果**
- 5 天完成约 80 个 API 端点（29 次提交，3月4-11日）
- 其中 13 次 fix 提交（占45%），说明 AI 代码有不少小错漏，但人在评审中及时捕获
- 对比传统开发：效率提升 2-4 倍

**为什么这次成功**
- 表结构设计清晰 → AI 编码质量高
- 人负责"定方向"，AI 负责"跑量"
- 及时评审，小步验证
- 评审中发现问题后更新 CLAUDE.md（编码规范），持续改进

**小结：设计到位 + 边界清晰 = AI 效率拉满**

### 1.2 案例二：A2A 适配器 — 一场代价惨痛的返工（10-13分钟）

**项目背景**
- 需要开发 A2A（Agent-to-Agent）协议适配器（aip-gateway 的 a2a-wrapper 插件）
- 这是一个相对陌生的领域

**发生了什么**
- 让 AI 根据 A2A 协议文档直接编码，没有先调研已有方案
- AI "高效"地产出了大量代码（16 天，46 次提交）
- 但因为自己对 A2A 不了解，无法判断 AI 的输出是否正确
- 深入查阅 A2A 官方网站后，发现有官方 Java SDK（a2aproject/a2a-java）
- 推倒重来，用 SDK 重新实现

**代价量化**
- 自研方案：141,026 行代码被删除
- SDK 重写：仅 1,786 行新增 — 代码量缩减 98.7%
- 时间：16 天 vs 3 天
- 根因：对领域不了解 + 没有先建立认知就动手 + 被 AI 的"高效产出"误导

**翻车现场补充（快速过 2-3 个小案例）**
- 以下是来自另一个项目（aip-portal）的教训：
- SQL 迁移版本号错误：AI 生成了 V2.2.14 而非 V2.2.13，因为没有先查看现有版本文件（来自 aip-portal 复盘记录）
- 修改已冻结的 Flyway 脚本：AI 修改了 V2.2.13__init.sql，因为担心预制模板与补丁不一致。正确做法是只新建补丁文件（来自 aip-portal 复盘记录）
- 权限过滤理解偏差：AI 对"前端可见性控制权限"策略理解不深，把应交给前端的逻辑保留在后端（来自 aip-portal 复盘记录）

**小结：方向错了，效率越高代价越大**

**第一部分收尾——引出的思考**
- 两个案例，截然不同的结果，背后有什么规律？
- 什么时候该信任 AI？什么时候必须人把关？
- 如何系统地让 AI 编程越来越高效？

## 第二部分：反思——从案例到洞察（10-15分钟）

> 目标：不是罗列经验，而是层层递进，最终引出一个关键问题，自然过渡到 Part 3

### 2.1 从案例中看到了什么（2-3分钟）

**回扣 Part 1 的两个案例**
- aip-server 成功的关键：人给了清晰的设计和边界 → AI 编码质量高
- A2A 失败的根因：人没有给方向 → AI 高效地跑错了路

**提炼出一个观察**
- AI 的效率是确定的，不确定的是人的引导
- 同一个 AI，同一个开发者，结果天壤之别
- 区别不在于 AI 的能力，而在于人怎么用它

**AI 的潜力与边界**
- 潜力：编码效率提升 2-4 倍是真实的，擅长重复性工作和模式化代码
- 边界：不了解领域背景时容易跑偏，缺乏全局视角，不会主动质疑你的设计
- 一句话：**AI 是最强执行者，但不是合格的架构师**

### 2.2 核心洞察：把最擅长的事交给 AI（3分钟）

**为什么从"最擅长"的开始？**
- 擅长 = 你能准确验收，质量有保障
- 这类工作对个人已没有成长价值，属于纯工作量堆叠
- 释放你的时间，投入到更有创造性的研究、设计、架构工作中

**与直觉相反**
- 很多人觉得"先把不擅长的交给 AI，自己做好擅长的"
- 但你不擅长的领域，你连 AI 的输出是否正确都无法判断

**aip-server 案例就是最好的例证**
- 表结构设计是我擅长的 → AI 编码我能准确评审
- 如果是我不了解的领域，80 个接口的输出质量根本无法把控

**角色转变**
- 过去：自己设计 + 自己编码 + 自己测试
- 现在：自己设计 + AI 编码 + 自己评审测试
- 从"写代码的人"变成"设计系统的人"

### 2.3 但这里有个陷阱：AI 会"PUA"你（3分钟）

**现象**
- AI 永远自信满满地给出方案，即使方向是错的
- 语气笃定、逻辑自洽，很容易被"说服"
- 不像人类同事会犹豫、会说"我不确定"

**翻车案例回扣**
- A2A 案例中，AI 确实"高效"地造了一个轮子，全程没有任何犹豫
- 权限过滤被保留时，AI 的解释听起来完全合理

**应对**
- 保持自己的判断力，不能被 AI 的"气势"带着走
- 对关键逻辑一定要人工验证，不要被表面的合理性蒙蔽

### 2.4 关键问题：怎么让 AI 稳定输出高质量代码？（2-3分钟）

**把前面的思考串起来**
- 擅长的事交给 AI → 但 AI 会 PUA 你 → 靠个人判断力够吗？
- aip-server 成功了，但那是靠设计到位 + 个人经验
- A2A 翻车了，靠的是事后补救

**问题升级**
- 单靠个人经验和判断力，不可持续，不可复制
- 团队里其他人怎么办？项目复杂了怎么办？
- 我们需要一套**系统化的方法**

**引出 Part 3**
- 有没有办法，让 AI 的输出质量不只是依赖个人的"手感"？
- 有。这就是 SDD 和 Harness 要解决的问题 →

## 第三部分：系统化提升 — SDD 与 Harness 实践（30-35分钟）

> 目标：用两个项目分别讲透 SDD 和 Harness，每个概念配一个专属案例
> SDD 用 aip-gateway 讲"规范先行"，Harness 用 aip-portal 讲"从混乱到精炼"

### 3.1 什么是 SDD（5分钟）

**用大白话解释**
- SDD = Specification-Driven Development（规范驱动开发）
- 不是新概念，但在 AI 编程中获得了全新的意义
- 传统开发中：规范是文档，人看文档写代码
- AI 编程中：规范是"AI 的行为边界"，直接决定 AI 输出质量

**一个类比**
- 就像给新员工 onboarding：
  - 没有规范 = 新员工自由发挥（结果不可控）
  - 有规范 = 新员工按章办事（结果可预期）
  - AI 就是那个"超级新员工"

**SDD 在 AI 编程中的三层体现**
- 项目级规范（CLAUDE.md、rules/）= 员工手册
- 任务级规范（Plan、PRD、Spec）= 任务说明书
- 会话级引导（对话中的约束和纠正）= 日常工作指导

### 3.2 用智能体消息网关讲 SDD 实践（12-15分钟）

**项目简介**
- 智能体消息网关（aip-gateway）中的 AMG 核心模块
- 技术栈：Spring WebFlux + R2DBC + Redis Reactive（响应式全链路）
- 在门户项目经验之后的新项目，有意识地从 Day 1 就系统化

**SDD 工作流：PRD → Spec → 审阅 → 编码**
- **PRD 阶段**：用 qoder 编写 PRD，人定义需求边界
- **Spec 阶段**：AI 辅助转化为技术方案，产出 4796 行详细设计文档
- **审阅阶段**：用 Claude Code 对 Spec 进行多轮审阅优化
  - 设计文档从 v1 迭代到 v1.9，经历 9 轮迭代
  - Git 提交序列清晰记录：feat:add-design-v1 → v2 → v3 → ... → 三轮审阅修复 → 实现
- **编码阶段**：Agent Team 创建智能体团队根据 Spec 自动编码

**三轮审阅发现的关键问题**
- 架构定位偏移：AMG 被设计成独立服务 → 修正为数字员工门户消息层（v1.6 重大调整）
- 技术栈兼容性：误用 Spring Boot 3.x API → 修正为 2.7.17，与现有网关一致
- 数据模型关系：三层关系表述不清 → 明确 de_catalog → digital_employee → ops_channel
- **核心观点**：审阅不是走形式，每次审阅都在防止"AI 高效跑偏"
- **回扣 Part 2**：这就是系统化地防止"AI PUA 你"

**成果**
- AMG 核心模块：59 个 Java 文件，仅 2 天完成
- 13,093 行代码新增，71 个文件变更
- CLAUDE.md 237 行，包含完整架构决策和 A2A 协议参考
- 没有经历混乱期，从一开始就是系统化的

**回扣 Part 1 的两个成功案例**
- aip-server：靠"经验 + 设计到位"（个人能力）
- aip-gateway：靠"系统化 SDD 流程"（方法论）
- 同样的结果，但 aip-gateway 的方式可复制、可团队化
- **金句：规范先行，代码自动**

### 3.3 什么是 Harness Engineering（3分钟）

**用大白话解释**
- Harness = 为 AI 搭建的"工作脚手架"
- 包括：规范文件、权限配置、自定义技能、工作流配置
- 目标：让 AI 在你的项目里工作得又快又准

**三大支柱**
- **上下文工程**：给 AI 正确的信息（rules、CLAUDE.md、知识库）
- **架构约束**：让 AI 在框架内工作（技术栈限制、目录结构约定）
- **熵管理**：防止混乱累积（定期清理、精简规则、代码评审）

**一句话：Harness 就是让你的项目对 AI 友好**

### 3.4 用智能体门户项目讲 Harness 的搭建与生长（12-15分钟）

**项目简介**
- 智能体门户（aip-portal）：企业级 Agent 管理平台
- 259 次提交，72 天开发，281 个 Java 文件
- 这个项目经历了 Harness 从无到有的完整演化

**Rules 的四阶段演化**（Harness 的核心实践）

| 阶段 | 规模 | 特点 | 问题 |
|------|------|------|------|
| Phase 1 初始 | 5487 行，20 个文件 | 什么都往里放 | Token 消耗大，AI 抓不住重点 |
| Phase 2 膨胀 | 24 个文件，~264KB | 持续堆砌，最大单文件 53KB | Token 爆炸，AI 经常忽略关键规则 |
| Phase 2.5 发现根因 | — | 发现 code-style 文件配了 `paths: **.java`，导致编辑任何 Java 文件时全部加载 | 每次编辑都注入大量无关上下文 |
| Phase 3 大清理 | 删掉 2714 行 | 确立核心原则，参考文档移至 docs/ | 效果明显改善 |
| Phase 4 精简 | 272 行（压缩率 95%） | 只保留关键约束 | 质量稳定，Token 可控 |

**核心经验**
- rules/ 是 token 预算管理，不是 wiki
- 规范不是越多越好，而是越精炼越好
- 带有 `paths` 的参考文档不应放在 rules/ 下，应该移到 docs/ 只保留索引
- 每次踩坑都是完善规范的机会（复盘驱动改进机制）

**Harness 最终形态：三层知识架构**
- 全局层（~/.claude/）：通用规则、偏好设置
- 项目层（项目 .claude/）：CLAUDE.md（23行）+ rules/（272行，3文件）+ docs/code-style/（838行，8文件，按需加载）
- 会话层（对话中）：临时约束、特定任务指导
- **设计理念**：核心约束自动加载 + 详细规范按需引用

**结果**
- AI 编码准确率稳定提升
- 重复性错误逐步减少
- 新会话也能快速上手，因为知识已经沉淀在项目里
- 259 次提交中，rules 相关提交集中在 4 月初（仅 9 天），之后趋于稳定

**两个项目的对比与启示**

| 维度 | aip-gateway（SDD 驱动） | aip-portal（Harness 演化） |
|------|----------------------|----------------------|
| 核心理念 | 规范先行，代码自动 | 从混乱中提炼秩序 |
| 规范建设 | Day 1 就建 Spec（v1→v1.9） | 踩坑后逐步建立（Phase 1→4） |
| 混乱期 | 基本没有 | 经历了约 2 个月 |
| rules/ 结构 | 集中在 CLAUDE.md（237行） | 分层 rules/ + docs/（272行 + 838行按需） |
| 开发效率 | AMG 核心模块 2 天 | 259 次提交，72 天 |

**启示**
- 有了方法论之后，新项目可以少走很多弯路
- "PRD → Spec → 审阅 → 编码" 是可复用的工作流

## 第四部分：工具链与实战技巧（10-15分钟）

> 目标：给听众可立即上手的工具、技巧和行动建议

### 4.1 Claude Code + cc-switch（3分钟）

**Claude Code 核心优势**
- 直接在终端工作，与项目代码零距离
- 支持读取、编辑、执行，全流程闭环

**cc-switch 的作用**
- 快速切换 Claude Code 配置（模型、权限等）
- 适应不同项目、不同阶段的需要

### 4.2 实战技巧（5分钟）

**技巧一：小步快跑，持续验证**
- AI 生成速度快，但错误积累也快，一次贪多反而更慢
- 设计 → 实现 → 测试，每完成一个模块就验证
- 遇到问题再"磨刀"，不要试图提前设计完美的体系

**技巧二：先建立认知，再让 AI 干活**
- 越是不了解的领域，越要先花时间调研
- 2 小时的调研可以避免 16天的返工（A2A 教训）
- 你的认知边界就是 AI 效率的上限
- 新领域：先让 AI 帮你建立认知框架，再进入编码

**技巧三：协作模式选择矩阵**

| 场景 | 推荐模式 | 说明 |
|------|----------|------|
| 明确的小任务 | 直接对话 | 给清楚需求，让 AI 直接做 |
| 复杂功能开发 | Plan 模式 | 先让 AI 规划，确认后再执行 |
| 多任务并行 | Worktree + Agent Team | 隔离开发，互不干扰 |
| 新领域探索 | 调研先行 | 先让 AI 帮你建认知 |
| 代码评审 | 只读模式 | 让 AI 读代码，人做决策 |

### 4.3 快速行动建议（3-5分钟）

**给新手的建议（60% 听众）**
- 从一个小项目或小模块开始
- 先用好 CLAUDE.md（写清楚技术栈和基本约定）
- 养成"设计 → AI 实现 → 验证"的习惯

**给进阶者的建议（30% 听众）**
- 开始搭建项目级 Harness（rules、Skills）
- 尝试 Worktree 并行开发
- 建立复盘习惯，持续优化规范

**给观望者的建议（10% 听众）**
- 不需要全面切换，先在一个小任务上试试
- 从"让 AI 写测试"或"让 AI 做代码评审"开始
- 感受到效率提升后自然会扩大使用范围

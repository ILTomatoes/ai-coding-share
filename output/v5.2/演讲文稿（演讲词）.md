## 第一部分：从真实项目说起（20-25 分钟）

先讲两个案例，一正一反。两个项目都用 AI 编程，但结局完全不一样。听完这两个案例，大家对 AI 编程的潜力和坑应该会有一个比较直观的感受。

### 1.1 案例一：aip-server 技能工具模块 — AI 编程的高光时刻

先说第一个案例，这是一个比较成功的经历。

背景是这样的：我们的 AI 智能平台后端服务，叫 aip-server，需要开发一个技能工具模块，在 logic/tool 目录下。这个模块涉及 5 个子模块——代码库管理、MCP 服务、OpenAPI、扩展插件、工作流，总共需要开发大约 80 个 API 端点。技术栈是 Spring Boot 2.7.18 加 MyBatis Plus 3.4.2，数据库用 PostgreSQL，跑在 Java 17 上。这个项目整体不小，光 Controller 就有 100 个，11 个业务模块。

当时的协作模式是这样的：我先设计好原型，然后自己设计表结构，AI 会参与讨论和评审。表结构定好之后，AI 来编码，我来评审和测试。评审过程中发现问题，我就把编码规范写到 CLAUDE.md 里，后面 AI 的代码质量就越来越好了。

说一下结果。5 天时间，完成了大约 80 个 API 端点，总共 29 次提交，从 3 月 4 号到 11 号。说实话这个速度我自己都挺意外的。不过有个数据值得注意：29 次提交里有 13 次 fix 提交，占 45%。这说明什么？说明 AI 的代码确实有不少小错漏，但是通过人工评审，这些问题都被及时捕获了。对比传统开发方式，效率大概提升了 2 到 4 倍。

那这次为什么成功呢？我总结了几点：表结构设计得清晰，AI 编码的时候边界就很明确；人负责定方向，AI 负责跑量；还有就是及时评审、小步验证。特别想提一点，评审中发现问题后我更新了 CLAUDE.md 里的编码规范，后续 AI 的代码质量确实有明显提升。这个"踩坑 -> 补规范"的模式，后面还会反复提到。

一句话总结这个案例：设计到位加上边界清晰，AI 的效率就能拉满。

### 1.2 案例二：A2A 适配器 — 一场代价惨痛的返工

然后说说第二个案例，这个就比较痛了。

背景是我们需要开发一个 A2A（Agent-to-Agent）协议的适配器，放在 aip-gateway 的 a2a-wrapper 插件里。A2A 这个东西当时对我来说是一个比较陌生的领域，没有深入研究过。

当时怎么做的呢？我直接让 AI 根据 A2A 协议文档去编码了。没有先调研有没有现成的方案，没有了解社区的实践，就让 AI 直接开干。AI 确实"高效"，16 天时间，46 次提交，产出了大量代码。但问题是什么呢？因为我自己对 A2A 不了解，我根本没法判断 AI 产出的代码到底对不对。

后来我自己花了时间去深入研究 A2A 官方网站，这才发现，人家有官方的 Java SDK，叫 a2aproject/a2a-java。早就有了。也就是说，我们花了 16 天，从零造了一个完全不必要的轮子。

发现之后当然是推倒重来，用官方 SDK 重新实现。大家猜猜重写用了多久？3 天。而且代码量差异惊人：自研方案产出了 141,026 行代码，全部被删除；用 SDK 重写只新增了 1,786 行。代码量缩减了 98.7%。

说白了，16 天的工作，大部分都是浪费。如果一开始先花 2 个小时调研一下，可能 3 天就搞定了。

除了这个大的返工，还有一些小的翻车案例也跟大家分享一下，都来自我们的另一个项目 aip-portal：

一个是 SQL 迁移版本号错误。AI 生成 Flyway 迁移脚本的时候，生成了 V2.2.14，但实际应该是 V2.2.13。为什么？因为它没有先去看一下现有的版本文件。AI 做事就是这样的，你让它生成版本号，它就生成一个看起来合理的，但不会主动去检查已有文件。

另一个是修改了已冻结的 Flyway 脚本。AI 创建补丁 SQL 的时候，担心预制模板和补丁不一致，就去改了已经冻结的 V2.2.13_init.sql。这个在已部署的环境上直接就报错了，因为文件摘要不一致。正确做法是只新建补丁文件，已冻结的脚本绝对不能动。

还有一个是权限过滤的理解偏差。AI 对"前端可见性控制权限"这个策略理解不够深，把应该交给前端的逻辑保留在了后端。听起来好像 AI 做得也没错，后端多检查一下权限有什么不好？但问题是这个权限策略本身就是面向前端的可见性控制，后端不应该额外做这些检查。AI 保留这些逻辑时的解释听起来完全合理，但缺失了业务上下文。

所以这个案例的总结就是：方向错了，效率越高代价越大。

说到这里，大家可能会想：两个案例，同样是用 AI 编程，一个 5 天搞定 80 个接口，一个 16 天造了个没用的轮子。差别在哪？什么时候该信任 AI？什么时候必须人来把关？有没有办法系统地让 AI 编程越来越高效？

这些问题，我们在后面的内容里慢慢展开。

## 第二部分：反思——从案例到洞察（10-15 分钟）

刚才讲了两个案例，一正一反。现在我们退一步，从这些案例里提炼一些规律性的东西。

### 2.1 从案例中看到了什么

先回顾一下：aip-server 成功的关键是什么？是人在前面给了清晰的设计和边界。AI 只需要在明确的框架里填代码，质量就有保障。A2A 失败的根因是什么？是人没有给方向。对领域不了解，没有先建立认知，就让 AI 直接干了。

所以这里有一个核心观察：AI 的效率是确定的，不确定的是人的引导。同一个 AI，同一个开发者，在不同的项目上结果天壤之别。区别不在于 AI 的能力，而在于人怎么用它。

说到 AI 的潜力和边界，我觉得可以这么概括：

潜力方面，编码效率提升 2 到 4 倍是真实的，不是吹的。AI 特别擅长重复性的工作和模式化的代码。而且这个趋势很明确，只会越来越强。

但边界也很清晰：AI 不了解你的领域背景时很容易跑偏；它缺乏全局视角，可能"局部最优但全局灾难"；最重要的是，AI 不会主动质疑你的设计，你让它做什么它就做什么。

一句话：AI 是最强执行者，但不是合格的架构师。

### 2.2 核心洞察：把最擅长的事交给 AI

接下来这个观点可能会跟大家的直觉有点反差。

很多人会觉得："我应该先把不擅长的事情交给 AI，自己做好擅长的部分。"听起来很合理对吧？但实际上恰恰相反，你应该先把最擅长的事情交给 AI。

为什么？三个原因：

第一，你擅长的事情，你能准确验收。你写了几年的 CRUD 接口，AI 写出来好不好你一眼就知道。质量有保障。

第二，这些你已经很擅长的工作，对你个人来说其实没有太多成长价值了。说白了就是纯工作量堆叠。

第三，把你从这些重复劳动中释放出来，你就有更多时间投入到研究、设计、架构这些更有创造性的工作上。

回看 aip-server 的案例，为什么成功？因为表结构设计是我擅长的，AI 编码我能准确评审。如果换成我不了解的领域，80 个接口的输出质量根本没法把控。

所以角色转变是这样的：过去我们是自己设计、自己编码、自己测试；现在是自己设计、AI 编码、自己评审测试。从"码农"变成了"系统架构师"——你负责画图纸，AI 负责盖楼。这个转变其实挺有意思的。

### 2.3 但这里有个陷阱：AI 会 PUA 你

但是，事情没有那么简单。AI 有一个特点，不知道大家有没有体会到——它永远自信满满。

就算方向是错的，AI 给方案的时候语气也是特别笃定，逻辑自洽，你很容易被它"说服"。它不像人类同事，人类遇到不确定的事情会犹豫，会说"我不确定"或者"我再想想"。AI 不会，它永远给你一个看起来很完美的方案。

回想一下 A2A 的案例：AI 全程没有任何犹豫，就自信地造了一个完整的轮子。如果它当时说一句"你确定要从零实现吗？可能有现成的 SDK"，这 16 天就不需要浪费了。权限过滤那个案例也是，AI 保留那些后端逻辑的时候，解释得头头是道，听起来完全合理，但它缺失了关键的业务上下文。

所以应对方式就是：保持自己的判断力，不能被 AI 的"气势"带着走。对关键逻辑一定要人工验证，不要被表面的合理性蒙蔽。

### 2.4 关键问题：怎么让 AI 稳定输出高质量代码？

把前面的思考串起来，我们面临这样一个链条：

应该把擅长的事交给 AI，但是 AI 会 PUA 你，那靠个人判断力够不够？

aip-server 确实成功了，但那是靠设计到位加上个人经验。A2A 翻车了，靠的是事后补救。单靠个人的经验和判断力，这种方式不可持续，也不可复制。你自己能搞定的项目还行，团队里其他人怎么办？项目复杂了怎么办？

所以我们需要一套系统化的方法。不是靠个人的"手感"，而是靠一套可复制、可团队化的流程和工具。

有没有这样的方法？有。这就是接下来要讲的 SDD 和 Harness 要解决的问题。

## 第三部分：系统化提升 — SDD 与 Harness 实践（30-35 分钟）

这一部分是今天最核心的内容。我会用两个概念和两个项目来讲，先讲 SDD 再讲 Harness，每个概念都会配一个专门的项目案例。

### 3.1 什么是 SDD

先说 SDD。SDD 全称是 Specification-Driven Development，规范驱动开发。

简单来说，SDD 是一种"先写说明书，再让 AI 写代码"的开发新范式。它的核心思想是：在写任何一行代码之前，先用自然语言把"做什么"和"为什么"定义清楚。这份"说明书"就叫 Spec（规格文档），它是人类开发者与 AI 智能体之间共同的唯一真理来源。

为什么需要 SDD？因为它要解决一个问题——"氛围编程"（Vibe Coding）。什么是氛围编程？就是凭感觉提需求，AI 猜着写，写错之后花几个小时修。大家回想一下 A2A 的案例，其实那就是典型的氛围编程——没有清晰的规范，就让 AI 直接干。SDD 就是为了终结这种模式而生的。

那 Spec 到底是什么？我打个比方，Spec 就像建筑的施工图纸。盖楼之前先画好图纸，图纸规定了每一层的结构、每一根钢筋的位置。施工队——也就是我们的 AI——按照图纸施工就行。它不是普通的需求文档，写完就压箱底了。Spec 是一份"活"的契约：它是人与 AI 协作的合同，规定了代码必须满足的行为和约束。代码只是 Spec 的"派生物"，Spec 才是核心。

一份高质量的 Spec 通常包含四个部分：目标与价值（解决什么问题）、上下文与约束（技术栈、依赖、性能标准）、功能逻辑（核心行为）、以及验证标准（怎么证明功能写对了）。注意这最后一条——验证标准。因为 Spec 里强制包含了测试标准，AI 生成的代码就自带了测试驱动开发（TDD）的基因。

SDD 的标准工作流分四个阶段：

第一阶段是定义（Specify），用自然语言写清楚"做什么"和"为什么"。

第二阶段是计划（Plan），确定技术架构和实现路径。

第三阶段是任务（Task），把大需求拆解成 AI 可以执行的、可验证的原子任务。

第四阶段是实现（Implement），AI 根据图纸干活，开发者负责审核和验证。

为什么我们要用 SDD？三个核心收益：

第一是可预测性。在 AI 动手之前，你已经确切知道它要改哪些文件、怎么改，不再是"开盲盒"。

第二是高质量代码。因为 Spec 里强制包含了测试标准，AI 生成的代码自带 TDD 基因，能大幅减少安全漏洞和业务逻辑错误。

第三是易于维护。一年后如果你想知道某个逻辑为什么存在，答案就在 Spec 里，而不是埋在成千上万行的代码堆里。

讲到这里，可能有人在想：写 Spec 本身要花时间，直接让 AI 写代码不是更快吗？这个疑问完全合理。我的经验是，取决于任务规模。改一个字段这种小事，直接让 AI 做就行，不用写 Spec。半天以内的功能，写个 5 到 10 行的迷你 Spec 对齐一下。超过一天的复杂功能，完整的 Spec 流程一定要走，因为返工的成本远超写 Spec 的成本。一个简单的判断标准：如果这个需求给新同事讲需要超过 5 分钟，就值得先写 Spec。还有一个隐性收益：Spec 写完之后，它本身就是最好的技术文档。以后新人接手、做 Code Review，都能从 Spec 里找到答案。

一句话总结 SDD：它把程序员从"码农"变成了"系统架构师"——你负责把意图表达成精准的 Spec，也就是画图纸，而 AI 负责把图纸变成代码，也就是盖楼。

### 3.2 用智能体消息网关讲 SDD 实践

概念讲完了，直接上案例。这个案例来自我们的智能体消息网关项目，叫 aip-gateway。

这个项目的 AMG 核心模块用的是全响应式技术栈：Spring WebFlux 加 R2DBC 加 Redis Reactive，整条链路都是响应式的。这个项目有个特殊背景：它是在门户项目之后启动的，也就是说，我已经有了前面的经验和教训。所以这次我是有意识地从第一天就系统化地来做。

来说一下 SDD 的四阶段工作流在这个项目里是怎么落地的：PRD 到 Spec 到审阅到编码，正好对应 Specify 到 Plan 到 Task 到 Implement 四个阶段。

首先是 PRD 阶段。用 qoder 编写 PRD，人来定义需求边界。这一步的核心是人把需求想清楚。

然后是 Spec 阶段。AI 辅助把 PRD 转化为技术方案，最终产出了一份 4796 行的详细设计文档。大家注意这个数字，4796 行的 Spec，这不是随便写写的。

接下来是审阅阶段，这也是最关键的一步。用 Claude Code 对 Spec 进行多轮审阅优化。设计文档从 v1 一直迭代到 v1.9，经历了 9 轮迭代。Git 提交记录非常清晰：从 feat:add-design-v1 一路到 v2、v3 直到 v1.9，中间还有三轮审阅修复，最后才是实现。

我来重点说说三轮审阅中发现了什么关键问题：

第一个是架构定位偏移。AMG 原本被设计成了一个独立服务，审阅中发现这个定位不对，它应该是数字员工门户的消息层。这个在 v1.6 做了重大调整。试想一下，如果这个定位问题没在审阅阶段发现，到编码的时候再改，代价会有多大？

第二个是技术栈兼容性问题。Spec 里误用了 Spring Boot 3.x 的 API，但我们的网关实际跑的是 2.7.17。这个如果不审阅，到编码的时候就等着编译错误吧。

第三个是数据模型关系。三层关系——de_catalog 到 digital_employee 到 ops_channel——表述不清楚，审阅中做了明确。

这里我想强调一个核心观点：审阅不是走形式。每一次审阅都在防止"AI 高效跑偏"。这其实就是系统化地在防止我们在第二部分说的"AI PUA 你"的问题。

最后是编码阶段。有了完善的 Spec 之后，Agent Team 创建智能体团队，根据 Spec 自动编码。

来看成果：AMG 核心模块 59 个 Java 文件，只用了 2 天就完成了。新增 13,093 行代码，71 个文件变更。CLAUDE.md 有 237 行，包含了完整的架构决策和 A2A 协议参考。最重要的是，这个项目从一开始就没有经历混乱期，全程都是系统化的。

这里可以回扣一下第一部分的成功案例。aip-server 是靠经验和设计到位成功的，那是个人的能力。aip-gateway 是靠系统化的 SDD 流程成功的，这是方法论。同样的好结果，但 aip-gateway 的方式可复制、可团队化。你不需要有我那样的经验，只要按照这个流程走，也能达到类似的效果。

一句话总结：规范先行，代码自动。

### 3.3 什么是 Harness Engineering

接下来讲第二个概念：Harness Engineering。

LangChain 在博客上提过一个公式，我觉得可能是 2026 年关于 AI 工程最精辟的一句话：Agent = Model + Harness。模型是马，Harness 是缰绳。光有马不行，你还得有一整套驾驭它的系统。

Harness 原意就是"马具"——马鞍、缰绳、嚼子那一整套东西。马速度快、力量大，但不套上缰绳就会跑偏。在 AI 领域，Harness 就是把 AI 这股力量安全地引导到你需要的方向上的整套约束系统。

Harness 有两大控制机制，形成一个闭环。

第一个是 Guides，前馈控制。在 AI 行动之前就设好规则，让它沿着正确的方向走。类比一下，就像高速公路的护栏——你不需要每秒去纠正，护栏在那里就行。CLAUDE.md、代码规范文档、架构决策记录、权限分级，这些都属于 Guides。在我们项目里，rules 目录里的规范文件就是典型的 Guides——每次会话自动加载，AI 不会忘记。

第二个是 Sensors，反馈控制。在 AI 做完事之后检测它做得对不对。自动化测试、代码 lint、CI 流水线，这些就是 Sensors。

这两个机制形成闭环：好的 Harness 等于 Guides 加 Sensors，前者防患于未然，后者亡羊补牢。每当你发现 Agent 犯了一个错误，就花时间去设计一个方案，让它永远不可能再犯同样的错误。这就是做 Harness 工程的日常。

还有一个核心原则要记住：约束先行。规则不是靠口头约定，是靠自动化来强制执行的。有了约束，速度才不会下降，架构才不会漂移。

Harness 的上下文供给分三层：项目级规范，就是 CLAUDE.md 和 rules 目录，相当于员工手册，所有会话共享、每次自动加载；任务级规范，就是 Plan、PRD、Spec 这些，相当于任务说明书；会话级引导，就是你在对话中实时补充的约束和纠正。这三层配合起来，就构成了完整的信息供给体系。

一句话：Harness 就是给 AI 烈马套上缰绳，让它在你的项目里跑得又快又准。

### 3.4 用智能体门户项目讲 Harness 的搭建与生长

概念说完了，上案例。这个案例来自我们的智能体门户项目 aip-portal。

aip-portal 是一个企业级的 Agent 管理平台，259 次提交，72 天的开发周期，281 个 Java 文件。这个项目的特别之处在于：它经历了 Harness 从无到有的完整演化过程。前面 aip-gateway 的案例是"规范先行"，这个项目则是"从混乱中提炼秩序"。

核心内容是 rules 目录的四阶段演化。

**Phase 1，初始阶段。** rules 目录里有 5487 行，20 个文件。特点就是什么都往里放——编码规范、架构约定、踩坑记录、参考文档，一股脑全塞进去。问题是 Token 消耗巨大，AI 根本抓不住重点，关键的规则被淹没在一堆信息里。

**Phase 2，膨胀阶段。** 文件增加到了 24 个，总大小大约 264KB。最大的单个文件有 53KB。为什么膨胀？因为踩坑了就加规则，越加越多。犯了错加一条规则，再犯错再加一条，形成了一个恶性循环。结果就是 Token 爆炸，AI 经常忽略关键规则。

**Phase 2.5，发现根因。** 这个阶段特别关键。我们发现了问题的根源：code-style 文件配置了 paths 匹配所有 Java 文件，意味着每次编辑任何 Java 文件的时候，整个 code-style 文件的内容都会被注入到 AI 的上下文里。你说这得多浪费 Token？

**Phase 3，大清理。** 删掉了 2714 行。确立了一个核心原则：rules 目录是 Token 预算管理，不是 wiki。把参考文档移到 docs 目录下，只保留索引。

**Phase 4，精炼阶段。** 最终 272 行，压缩率 95%。只保留 AI 必须知道的约束。

来看一下最终形态，这是一个三层知识架构：

最上面是 CLAUDE.md，只有 23 行，纯项目导航。中间是 rules 目录，272 行，3 个文件，核心约束自动加载。下面是 docs/code-style 目录，838 行，8 个文件，按需加载。

设计理念很清楚：核心约束每次会话都自动加载，详细规范按需引用，AI 需要的时候自己去查。这样既保证了 AI 能看到关键规则，又不会因为信息过载而忽略重点。

几个核心经验：

rules 目录是 Token 预算管理，不是 wiki。每一行都要有存在的价值。

规范不是越多越好，而是越精炼越好。少即是多。

每次踩坑都是完善规范的机会。这里有一个核心原则：每当你发现 AI 犯了一个错误，就花时间去设计一个方案，让它永远不可能再犯同样的错误。这就是我们前面说的 Harness 闭环——Guides 防患于未然，Sensors 亡羊补牢。但加完之后要定期回头看，该删的删，该精简的精简。

还有一个实操经验：带有 paths 配置的参考文档不应该放在 rules 目录下，应该移到 docs 目录只保留索引。因为 paths 配置意味着每次编辑匹配文件时都会加载这些内容，很容易造成 Token 浪费。

最后来对比一下这两个项目：

aip-gateway 用的是 SDD 驱动的模式，核心理念是"规范先行，代码自动"。从第一天就建 Spec，从 v1 迭代到 v1.9，基本没有混乱期。

aip-portal 用的是 Harness 演化的模式，核心理念是"从混乱中提炼秩序"。踩坑后逐步建立规范，经历了大约 2 个月的混乱期。

aip-gateway 的 AMG 核心模块 2 天完成，aip-portal 是 259 次提交、72 天。

但重点不是谁快谁慢，而是启示：有了方法论之后，新项目可以少走很多弯路。"PRD 到 Spec 到审阅到编码"这个工作流是可复用的。你不需要自己经历 72 天的混乱期，可以直接套用这套方法论。

## 第四部分：工具链与实战技巧（10-15 分钟）

理论和方法论讲了不少，最后这部分来点实用的——工具和技巧，听完就能上手。

### 4.1 工具介绍：Claude Code 和 cc-switch

先说工具。我主要用的 AI 编程工具是 Claude Code，它是一个终端工具。核心优势就是直接在终端里工作，和项目代码零距离。支持读取文件、编辑代码、执行命令，全流程闭环。而且它会自动加载项目的 CLAUDE.md 和 rules 目录，理解你的项目上下文。

另外我搭配使用了一个叫 cc-switch 的工具，用来快速切换 Claude Code 的配置。比如不同项目需要不同的模型或者权限设置，用 cc-switch 一键切换就行。复杂任务用强模型，简单任务用快模型，挺方便的。

### 4.2 实战技巧

再说三个我觉得最重要的实战技巧。

**技巧一：小步快跑，持续验证。**

AI 生成代码的速度很快，但错误积累得也很快。一次贪多反而更慢。正确的做法是：设计、实现、测试，每完成一个模块就验证一下。遇到问题再"磨刀"，不要试图提前设计一套完美的体系。我们 aip-portal 早期就吃过这个亏，一次做一个大模块，结果出了问题要批量返工。

**技巧二：先建立认知，再让 AI 干活。**

这个是从 A2A 那次惨痛教训中总结出来的。越是不了解的领域，越要先花时间调研。你的认知边界就是 AI 效率的上限。A2A 那次，如果先花 2 小时调研，就不需要 16 天的返工。

实践建议：如果是新领域，先让 AI 帮你建立认知框架，梳理一下领域知识，然后再进入编码。如果是你熟悉的领域，可以直接进入设计-实现的流程。

**技巧三：根据场景选择协作模式。**

不同场景适合不同的模式，我总结了一个简单的矩阵：

如果是明确的小任务，直接对话就行，给清楚需求让 AI 做。

如果是复杂功能开发，用 Plan 模式，先让 AI 规划，确认后再执行。

如果需要多任务并行，用 Worktree 加 Agent Team，隔离开发互不干扰。

如果是新领域探索，先调研，别急着写代码。

如果是代码评审，用只读模式，让 AI 读代码，人做决策。

### 4.3 快速行动建议

最后，根据大家不同的阶段，给一些可以马上行动的建议。

**如果你刚开始实践 AI 编程**，也就是在座大部分同学：

第一步，从一个小项目或者小模块开始，不要上来就整个项目都用 AI。

第二步，先写好一个 CLAUDE.md 文件。不用很复杂，写清楚技术栈、项目结构、核心约定就行。比如用 Spring Boot 2.7 加 MyBatis Plus，统一分页查询用什么类，API 返回统一用什么包装，数据库迁移怎么做。就这些基本信息，已经能让 AI 的输出质量好很多。

第三步，养成"设计、AI 实现、验证"的习惯。每次都先想清楚要什么，再让 AI 做，做完自己检查。

**如果你已经有一定经验了**，可以开始搭建项目级的 Harness。写一个 20 到 30 行的 CLAUDE.md 做项目导航，建 rules 目录放架构约束和编码规范索引，详细规范放到 docs 下按需引用。另外可以试试 Worktree 做并行开发，我们 aip-portal 项目用了 24 个 worktree。还有就是要建立复盘习惯：踩了坑就补规范，定期精简。

**如果你还在观望**，也没关系。不需要全面切换，先在一个小任务上试试。比如让 AI 帮你写单元测试、做代码评审、写 CRUD 接口。这些场景简单、边界清晰，适合入门。感受到效率提升之后，你自然会想扩大使用范围。

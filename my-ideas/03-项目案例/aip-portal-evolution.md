# Claude Code 实践总结：aip-portal 项目

> 本文档总结了在 aip-portal 项目中使用 Claude Code 的实践经验，包括知识体系搭建、工作流优化、以及后续改进方向。

---

## 一、知识体系的三层架构

Claude Code 的知识传递依赖三个层级，每层职责不同、生命周期不同：

```
┌─────────────────────────────────────────────────┐
│  全局层 (~/.claude/CLAUDE.md)                    │
│  作用域：所有项目                                 │
│  内容：跨项目的个人偏好和通用规则                   │
│  生命周期：长期稳定，偶尔新增                       │
│                                                  │
│  本项目实际内容：                                  │
│  - Git 提交规范（中文 type: 格式，禁止署名）       │
│  - Shell 环境重定向规范（/dev/null vs nul）        │
│  - 配置级别默认规则（全局 vs 项目）                 │
│  - 排除 Claude Code 本地配置文件                   │
└────────────────────┬────────────────────────────┘
                     │
┌────────────────────▼────────────────────────────┐
│  项目层 (CLAUDE.md + .claude/rules/)             │
│  作用域：当前项目                                 │
│  内容：项目架构、编码规范、测试规范                 │
│  生命周期：随项目迭代更新                          │
│  特点：每次对话全部加载，每行都在消耗 token         │
│                                                  │
│  本项目实际结构：                                  │
│  CLAUDE.md           → 文档索引 + 常用命令         │
│  rules/ARCHITECTURE  → 架构全景                   │
│  rules/CODE-STYLE    → 规范索引                   │
│  rules/code-style/01~08 → 编码规范专题            │
│  rules/testing       → 测试规范                   │
└────────────────────┬────────────────────────────┘
                     │
┌────────────────────▼────────────────────────────┐
│  会话层 (memory/)                                │
│  作用域：跨会话的经验积累                          │
│  内容：复盘经验、参考索引、项目上下文、用户偏好      │
│  生命周期：持续积累，定期清理                       │
│  特点：按需检索，不自动全部加载                     │
│                                                  │
│  本项目实际使用：                                  │
│  MEMORY.md           → 索引文件                   │
│  feedback/retrospect → 复盘经验教训                │
│  reference/          → 外部资源索引                │
└─────────────────────────────────────────────────┘
```

### 设计原则

| 原则 | 说明 |
|------|------|
| **信息不重复** | 同一条规则只存在于一个层级，其他层级引用 |
| **层级边界清晰** | 全局放"我是谁/我怎么工作"，项目放"这个项目怎么做"，会话放"上次踩了什么坑" |
| **Token 有意识** | 项目层每次全部加载，所以必须有存在价值；会话层按需检索，可以更自由 |

## 二、rules/ 配置的演变历程

### Phase 1：初始搭建（03-25 ~ 03-26）

项目刚开始接触 Claude Code 时，配置比较原始：

- ARCHITECTURE.md 是架构描述，内容较粗糙
- CODE-STYLE.md 是单文件 819 行的大杂烩，包含所有编码规范
- 没有测试规范

### Phase 2：功能驱动扩张（03-27 ~ 03-31）

随着功能开发推进，每做新功能就往 rules/ 加文档：

```
rules/
├── ARCHITECTURE.md                              # 架构
├── CODE-STYLE.md                                # 819行大文件
├── api-design/                                  # API 设计文档
│   ├── 20260324-feature-digital-employee-api.md
│   ├── 20260326-sys-menu-design.md
│   ├── 20260327-sys-role-design.md
│   └── 20260330-intelligent-assistant-api.md
├── design/                                      # 功能设计文档
│   ├── 20260327-intelligent-assistant-design.md  # 1706行
│   └── 20260331-sql-migration-starter-design.md  # 465行
├── exec-plans/                                  # 执行计划
│   ├── completed/                               # 已完成的计划（数个 300~1000行）
│   └── ...
└── generated/                                   # 自动生成文档
    ├── 20260324-feature-digital-employee-schema.md
    └── db-schema-v1.md                          # 454行
```

**问题暴露**：
- 设计文档（1706行助理设计）和执行计划（1000+行角色实现）大量占用 token
- 每次对话都加载所有文件，包括已完成的历史计划
- 规范文档和设计文档混在一起，AI 检索效率低

### Phase 3：关键转折——精简重构（04-01）

这是一次关键决策，确立了 rules/ 的核心原则：**rules/ 是给 AI 的参考，不是给人看的 wiki**。

**拆分**（+1175 行，但结构更清晰）：
- CODE-STYLE.md 819行 → 8 个专题文件 + 1 个索引文件
- 每个文件聚焦一个主题（命名、注解、数据库、API 设计等）

**大清理**（-6278 行）：
- 删除已完成的执行计划（completed/ 目录）
- 设计文档从 rules/ 移除，仅保留在 docs/
- 删除自动生成的 db-schema（可再生的文档不占 rules/ 空间）
- API 设计文档从 rules/ 移到 docs/api-design/

**最终结构**：
```
rules/
├── ARCHITECTURE.md     # 架构全景（1份，稳定）
├── CODE-STYLE.md       # 规范索引（指向 code-style/）
├── code-style/         # 8个专题文件
│   ├── 01-naming.md
│   ├── 02-annotations.md
│   ├── 03-database.md
│   ├── 04-api-design.md
│   ├── 05-comments-logging.md
│   ├── 06-vo-di.md
│   ├── 07-date-constants.md
│   └── 08-pagination.md
└── testing.md          # 测试规范
```

### Phase 4：稳定维护（04-02 ~ 至今）

精简后进入稳定期：
- 仅在架构变更时更新 ARCHITECTURE.md（如新增 operations 模块、新增 icon 字段）
- 编码规范文件几乎不变
- 新增了 Karate 测试规范（testing.md）

**经验**：好的结构不是一步到位的，而是「膨胀 → 识别问题 → 大清理 → 稳定」的自然循环。

## 三、记忆框架的搭建

### 记忆的类型与使用

Claude Code 的记忆系统支持四种类型，本项目实际使用了两种：

| 类型 | 使用情况 | 说明 |
|------|----------|------|
| **feedback** | 3 条 | 复盘经验教训，是最有价值的记忆类型 |
| **reference** | 1 条 | rules 文件管理规范，定义目录规划和生命周期 |
| project | 未使用 | 可记录项目里程碑、团队决策等 |
| user | 未使用 | 可记录技术背景、偏好工作方式 |

### 记忆的生产流程

```
完成任务
    │
    ▼
用户触发复盘（/retrospect 或"复盘"）
    │
    ▼
三问反思：
  1. 做对了什么？
  2. 做错了什么？
  3. 如果重新做会怎么做？
    │
    ▼
提炼可复用经验（跨任务、跨场景的）
    │  ┌──────────────────────────────────┐
    │  │ 过滤标准：                        │
    │  │ ✗ 一次性技术细节（已在代码中）      │
    │  │ ✗ 可从代码推导的信息               │
    │  │ ✓ 踩坑原因和预防方法               │
    │  │ ✓ 工作流改进方向                   │
    │  └──────────────────────────────────┘
    ▼
保存到 memory/feedback-retrospect-{date}.md
    │  格式：规则 + Why（为什么）+ How to apply（何时用）
    ▼
更新 MEMORY.md 索引（一行一条，带链接）
```

### 实际记忆示例

**feedback 类型**（来自 2026-04-08 复盘）：

```markdown
## 做错了
- SQL 迁移版本号命名错误（V2.2.14 而非 V2.2.13）。
  Why: 没有在开始前确认当前开发版本号，依赖了错误假设。
  How to apply: 涉及版本号命名时，先查看 resources/db/migration/postgresql/ 下已有的文件。
```

**reference 类型**（rules 文件管理规范）：

```markdown
## 管理规则
| 规则 | 说明 |
|------|------|
| 只放 Claude Code 需要的 | rules/ 是给 AI 的参考，不是给人看的 wiki |
| 一份文件一个主题 | 避免大杂烩文件，拆分后 AI 检索更精准 |
| 及时清理 | 过时的文件及时删除，每一行都在消耗 token |
```

## 四、工作流优化轨迹

### 4.1 从"直接开干"到"先规划再做"

**早期**：git 记录显示连续的 fix、fix、fix 模式，说明边做边改。

**后来**：逐步形成了 Plan 模式的工作流：
1. Plan 阶段先探索代码结构、识别可复用组件
2. 对关键不确定信息先向用户确认（版本号、字段来源等）
3. 确认方案后再进入实现阶段
4. 实现后运行编译验证

### 4.2 文档分层意识

| 时期 | 做法问题 | 改进后 |
|------|----------|--------|
| 早期 | 所有文档堆在 rules/ | 设计文档放 docs/，规范放 rules/ |
| 早期 | 人看的和 AI 看的混在一起 | rules/ 只给 AI 看，项目 wiki 给人看 |
| 早期 | CODE-STYLE 单文件 819 行 | 拆成 8 个主题文件 + 索引，按专题检索 |

### 4.3 经验制度化

从错误中提炼规则是最高效的知识积累方式：

| 踩坑场景 | 沉淀结果 |
|----------|----------|
| Git Bash 中 `> nul` 创建垃圾文件 | 全局 CLAUDE.md 的 Shell 重定向规范 |
| Claude 默认加 Co-Authored-By | 全局 CLAUDE.md 的 Git 提交规范 |
| SQL 版本号命名错误 | memory feedback 记录 |
| 权限控制理解偏差 | memory feedback + ARCHITECTURE.md 更新 |

### 4.4 复盘驱动的持续改进

通过 `/retrospect` 命令触发结构化复盘，将隐性的「下次注意」变成显性的知识记录。每条经验包含：
- **What**：发生了什么
- **Why**：根本原因是什么
- **How to apply**：未来何时/如何应用

## 五、继续优化工作流的具体建议

### 5.1 Hook 自动化

Claude Code 支持 Hooks 机制，可以在特定事件前后自动执行命令。

**建议配置**（写入 `.claude/settings.json`）：

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "cd D:/GitCode/gitlab/aip-portal && mvn compile -q -DskipTests 2>&1 | tail -5",
        "description": "Java 文件修改后自动编译验证"
      }
    ],
    "PreCommit": [
      {
        "command": "cd D:/GitCode/gitlab/aip-portal && mvn test-compile -q 2>&1 | tail -5",
        "description": "提交前验证测试编译"
      }
    ]
  }
}
```

**价值**：代码修改后自动验证编译，不用手动提醒 Claude 执行编译检查。

### 5.2 Skills 工作流链

已安装的 superpowers 技能包提供了完整的开发工作流，建议按以下链路使用：

```
需求提出
    │
    ▼
/superpowers:brainstorming     ← 探索需求和设计方案
    │
    ▼
/superpowers:writing-plans     ← 编写实施计划
    │
    ▼
/superpowers:executing-plans   ← 执行计划（带审查检查点）
    │
    ▼
/superpowers:verification     ← 完成前验证（运行测试、编译）
    │
    ▼
/superpowers:requesting-code-review  ← 请求代码审查
    │
    ▼
/retrospect                   ← 复盘，保存经验
```

**价值**：每个环节有明确的检查标准，避免跳步导致的返工。

### 5.3 补充记忆类型

当前只使用了 feedback 和 reference 两种记忆类型，建议补充：

**user 类型**（记录开发者背景）：

```markdown
---
name: 开发者背景
description: 用户的技术背景和工作偏好
type: user
---

- 后端为主，Java/SQL 是主要技术栈
- 前端由其他团队成员负责，后端不需要深究前端细节
- 偏好简洁的沟通风格，不需要过多的解释和总结
```

**project 类型**（记录项目上下文）：

```markdown
---
name: 项目里程碑
description: 项目重要节点和决策
type: project
---

- 2026-04-09: 系统初始化器完成，admin 用户和默认工作空间自动创建
- 2026-04-08: 助理模块新增 icon 属性，Karate 测试框架搭建完成
- 2026-04-01: rules/ 大精简，删除 6000+ 行冗余内容
```

### 5.4 定期复盘制度化

当前复盘由用户手动触发，建议建立定期复盘节奏：

| 时机 | 触发方式 | 复盘重点 |
|------|----------|----------|
| 功能完成后 | `/retrospect` | 实现过程、踩坑经验 |
| 每周 | Cron 定时提醒 | 本周效率、工作流改进 |
| 规则更新后 | 手动 | 规则是否需要精简或补充 |

### 5.5 Team 多 Agent 协作

对于复杂任务，可以使用 Team 模式并行处理：

```
用户提出复杂需求
    │
    ▼
创建 Team（TeamCreate）
    │
    ├─→ Agent A：探索代码库（Explore 类型，只读）
    ├─→ Agent B：编写计划（Plan 类型，只读）
    └─→ Agent C：研究外部文档
    │
    ▼
汇总结果，制定实施计划
    │
    ├─→ Agent D：实现后端代码
    └─→ Agent E：编写测试用例
    │
    ▼
汇总结果，代码审查
```

**适用场景**：大型功能开发、多模块重构、需要同时查阅多方资料的调研任务。

### 5.6 rules/ 维护清单

建议每次有较大变更后，检查以下清单：

- [ ] ARCHITECTURE.md 的模块列表是否需要更新？
- [ ] 是否有新引入的技术栈需要补充？
- [ ] 是否有已完成的文档需要从 rules/ 移除？
- [ ] 编码规范是否需要新增条目？
- [ ] 数据库表是否有变更需要反映在文档中？

## 六、关键经验总结

1. **rules/ 是 token 预算管理**：每个文件都要有存在价值，AI 不需要的信息就是噪音
2. **记忆系统是经验制度化**：不复盘等于白做，不保存等于白复盘
3. **三层架构各司其职**：全局放偏好，项目放规范，会话放经验，不要把信息放错层级
4. **先有再优化**：早期大胆堆内容，有积累后再精简，比一开始追求完美更实际
5. **从错误中提炼规则**：每次踩坑都是完善知识体系的机会
6. **文档有生命周期**：创建 → 迭代 → 稳定 → 过时（更新或删除），不让过期文档占用 token

# Java 开发中的 AI 协作编程实践 — 大纲版

> 线上会议 | 总时长约 90 分钟

---

## 开场

- 不是正式讲座，是真实项目经历的分享——有成功有翻车
- 材料来自三个 Java 项目：aip-server、aip-portal、aip-gateway
- 不同阶段的人都能带走有用的东西

---

## 第一部分：从真实项目说起（20-25 分钟）

### 1.1 aip-server 技能工具模块 — 成功案例

**背景**：5 个子模块，约 80 个 API 端点。Spring Boot 2.7 + MyBatis Plus + PostgreSQL，Java 17。

**协作模式**：人设计原型和表结构 → AI 编码 → 人评审测试 → 问题写入 CLAUDE.md → 后续质量提升。

**结果**：5 天完成 80 个 API，29 次提交。13 次 fix 提交（45%），AI 有错漏但人工评审能捕获。效率提升 2-4 倍。

**成功关键**：表结构清晰、边界明确；人定方向、AI 跑量；及时评审、小步验证；踩坑→补规范。

### 1.2 A2A 适配器 — 失败案例

**背景**：开发 A2A 协议适配器，领域陌生，未调研直接让 AI 开干。

**结果**：16 天、46 次提交，产出大量代码。后发现官方有 Java SDK（a2aproject/a2a-java）。

**推倒重来**：用 SDK 重写仅 3 天。代码从 141,026 行 → 1,786 行，缩减 98.7%。

**教训**：如果先花 2 小时调研，就不需要 16 天返工。

**aip-portal 小翻车**：
- SQL 迁移版本号错误（没检查已有文件）
- 修改已冻结的 Flyway 脚本（部署环境报错）
- 权限过滤理解偏差（缺失业务上下文）

**一句话**：方向错了，效率越高代价越大。

## 第二部分：反思——从案例到洞察（10-15 分钟）

### 2.1 核心观察

- 成功项目：人给了清晰设计和边界，AI 在框架内填代码
- 失败项目：人没给方向，对领域不了解就让 AI 直接干
- **AI 的效率是确定的，不确定的是人的引导**
- AI 是最强执行者，但不是合格的架构师

### 2.2 核心洞察：把最擅长的事交给 AI

- 你擅长 → 你能准确验收 → 质量有保障
- 擅长的工作对你没成长价值 → 纯工作量堆叠
- 释放时间 → 投入设计、架构等创造性工作
- 角色转变：从"码农"到"系统架构师"——你画图纸，AI 盖楼

### 2.3 陷阱：AI 会 PUA 你

- AI 永远自信满满，方向错了也语气笃定、逻辑自洽
- A2A 案例中 AI 全程没犹豫，自信造了完整轮子
- 应对：保持判断力，关键逻辑必须人工验证

### 2.4 问题：靠个人判断力不够

- aip-server 成功靠个人经验，A2A 翻车靠事后补救
- 不可持续、不可复制
- 需要系统化方法 → SDD 和 Harness

## 第三部分：系统化提升 — SDD 与 Harness（30-35 分钟）

### 3.1 SDD（Specification-Driven Development）

**定义**：先写说明书（Spec），再让 AI 写代码。Spec 是人与 AI 之间唯一的真理来源。

**要解决的问题**：终结"氛围编程"——凭感觉提需求，AI 猜着写，写错花时间修。

**Spec 是什么**：像建筑图纸——规定结构，AI 按图施工。是一份"活"的契约，代码是 Spec 的派生物。

**Spec 四要素**：目标与价值 | 上下文与约束 | 功能逻辑 | 验证标准（自带 TDD 基因）

**工作流四阶段**：定义（Specify）→ 计划（Plan）→ 任务（Task）→ 实现（Implement）

**三个收益**：可预测性 | 高质量代码 | 易于维护

**何时写 Spec**：小事直接做；半天功能写迷你 Spec；超一天走完整流程。标准：给新同事讲超 5 分钟就值得写。

### 3.2 aip-gateway — SDD 实践

**背景**：AMG 核心模块，全响应式技术栈，从第一天就系统化。

**四阶段落地**：PRD → Spec（4796 行）→ 审阅（v1 迭代到 v1.9，9 轮）→ 编码

**三轮审阅发现的关键问题**：
- 架构定位偏移（独立服务 → 应该是消息层）
- 技术栈兼容性（误用 Spring Boot 3.x API，实际是 2.7.17）
- 数据模型关系表述不清

**成果**：59 个 Java 文件，2 天完成。新增 13,093 行，71 个文件变更。从第一天没有混乱期。

**对比 aip-server**：个人经验成功 vs 系统化流程成功，后者可复制、可团队化。

### 3.3 Harness Engineering

**公式**：Agent = Model + Harness。模型是马，Harness 是缰绳。

**两大机制**：
- **Guides（前馈控制）**：行动前设好规则，像高速公路护栏。CLAUDE.md、规范文档、架构决策记录。
- **Sensors（反馈控制）**：做完后检测对不对。自动化测试、lint、CI。

**核心闭环**：每发现一个错误 → 设计方案让它永远不再犯。

**约束先行**：规则靠自动化强制执行，不靠口头约定。

**上下文三层供给**：项目级（CLAUDE.md + rules，自动加载）| 任务级（Plan/PRD/Spec）| 会话级（实时补充）

### 3.4 aip-portal — Harness 从混乱到秩序

**背景**：259 次提交、72 天，经历了 Harness 从无到有的完整演化。

**rules 目录四阶段演化**：

| 阶段 | 规模 | 特点 |
|------|------|------|
| Phase 1 初始 | 5487 行/20 文件 | 什么都往里放 |
| Phase 2 膨胀 | 24 文件/264KB | 踩坑加规则，恶性循环 |
| Phase 2.5 根因 | — | code-style 的 paths 匹配所有 Java，每次加载全部内容 |
| Phase 3 清理 | 删 2714 行 | 确立原则：rules 是 Token 预算管理，不是 wiki |
| Phase 4 精炼 | 272 行 | 压缩率 95% |

**最终三层知识架构**：
- CLAUDE.md：23 行，项目导航
- rules：272 行，3 文件，核心约束自动加载
- docs/code-style：838 行，8 文件，按需加载

**两个项目对比**：
- aip-gateway：SDD 驱动，"规范先行"，2 天完成，无混乱期
- aip-portal：Harness 演化，"从混乱中提炼秩序"，约 2 个月混乱期
- 启示：方法论可复用，不需要自己经历 72 天混乱

## 第四部分：工具链与实战技巧（10-15 分钟）

### 4.1 工具

- **Claude Code**：终端工具，和项目零距离，自动加载 CLAUDE.md 和 rules
- **cc-switch**：一键切换配置（模型/权限）

### 4.2 三个技巧

1. **小步快跑，持续验证**：设计→实现→测试，每模块验证。不提前设计完美体系。
2. **先建立认知，再让 AI 干活**：认知边界 = AI 效率上限。新领域先调研。
3. **按场景选模式**：小任务→直接对话 | 复杂功能→Plan 模式 | 并行→Worktree+Team | 新领域→先调研 | 评审→只读模式

### 4.3 快速行动建议

- **刚开始**：从小模块开始 → 写好 CLAUDE.md → 养成"设计→AI 实现→验证"习惯
- **有经验**：搭建项目级 Harness → 试试 Worktree 并行 → 建立复盘习惯
- **在观望**：先在小任务试——写单元测试、做代码评审、写 CRUD，感受到提升自然会扩大

## 收尾

四句话总结：
1. AI 的效率是确定的，不确定的是人的引导
2. 把最擅长的事交给 AI，释放时间做更有价值的事
3. 用 SDD + Harness 系统化驾驭 AI，不靠手感
4. 从"码农"变成"系统架构师"——你画图纸，AI 盖楼

**如果只记住一句：规范先行，代码自动。**

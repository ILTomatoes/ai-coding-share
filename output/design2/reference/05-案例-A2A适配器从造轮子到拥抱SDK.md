# 案例：A2A 协议适配器——从"自己造轮子"到"拥抱官方 SDK"

> 一个关于"不了解就开干"导致大量返工的真实案例

---

## 一、项目背景

**项目**: aip-gateway（基于 Apache ShenYu 的 AI 服务网关）
**任务**: 为网关开发 A2A（Agent-to-Agent）协议适配器插件，让网关能代理 A2A 协议的智能体通信
**开发方式**: 完全使用 Claude Code 辅助编程

---

## 二、时间线：从 0 到"推倒重来"

### 阶段一：自信开干，完全手写（01-20）

2026 年 1 月 20 日，一天之内搞了两个协议适配器：
- 上午：AG-UI 协议适配器（3136 行）
- 下午：A2A 协议适配器（3840 行）

**当时的做法**：
- 没有深入阅读 A2A 官方协议规范
- 没有搜索是否已有官方 SDK
- 让 AI 根据协议文档概要直接生成完整实现
- 完全手写了 JSON-RPC Handler、数据模型、错误处理等所有组件

**为什么这么快就"完成"了**：
- AI 能快速根据协议描述生成看起来合理的代码
- 代码结构完整，编译能过
- 但**协议实现的正确性完全没有验证**

### 阶段三：醒悟——发现官方 SDK（01-28 ~ 01-29）

**2026-01-28 是转折点**。

一个关键的 Claude Code 会话标题是：**"创建 A2A Wrapper 插件替代旧适配器"**

会话 prompt 开头：
> "创建新的 a2a-wrapper ShenYu 插件，完全替代现有的 a2a-protocol-adapter 插件。新插件直接集成 a2a-java SDK..."

**发生了什么**：
1. 终于深入阅读了 A2A 官方 GitHub 仓库
2. 发现官方提供了完整的 `a2a-java` SDK（`io.github.a2asdk`）
3. SDK 包含了所有需要的核心组件：`AgentCard`、`Task`、`AgentExecutor`、`RestHandler`...
4. SDK 还有 Spring WebFlux 的示例代码

**然后做了什么**：
- 把整个 `a2a-java` 官方仓库克隆到本地作为参考（135,226 行！）
- 用 **9 个 Claude Code 会话** 专门学习 SDK 源码
- 在 `spring-webflux-a2a-adapter` 示例项目上反复调试

### 阶段四：基于 SDK 重写，一周搞定（01-29 ~ 02-05）

经过一周的学习和调试，2026-02-05 是决定性的一天——**15 个提交**：

**核心提交 `60883e4`**：
- **删除** 旧 `a2a-protocol-adapter` 模块（**8,618 行删除**）
- **创建** 新 `a2a-wrapper` 模块（**2,114 行新增**）
- 110 个文件变更
- 直接 import 官方 SDK：

```java
import io.a2a.spec.AgentCard;
import io.a2a.spec.Task;
import io.a2a.server.agentexecution.AgentExecutor;
import io.a2a.server.requesthandlers.DefaultRequestHandler;
import io.a2a.transport.rest.handler.RestHandler;
```

**新方案对比旧方案**：

| 维度 | 旧方案（手写） | 新方案（SDK） |
|------|--------------|--------------|
| 代码量 | ~7,600 行 | ~2,400 行 |
| 开发时间 | 9 天（含返工） | 9 天（含学习） |
| 协议正确性 | 多处不一致 | 由 SDK 保证 |
| 维护成本 | 高（需跟进行规变化） | 低（SDK 更新即可） |
| 核心实现 | 自己写的 JsonRpcHandler | SDK 的 RestHandler |

## 四、根因分析

### 为什么会这样？

```
不了解 A2A 协议
    → 没有阅读官方文档
        → 不知道有官方 SDK
            → 让 AI "自由发挥"实现协议
                → AI 生成的代码结构完整但细节不对
                    → 不断修补
                        → 越补越偏
                            → 最终推倒重来
```

### 三个关键错误

| 错误 | 描述 | 后果 |
|------|------|------|
| **没有先调研** | 没有深入阅读 A2A 官方 GitHub 仓库 | 不知道有官方 SDK，白白手写了 7600 行 |
| **信任 AI 输出** | AI 生成的协议实现看起来合理，但与规范有多处不一致 | 规范符合性修复花了 7 天 |
| **缺少验证手段** | 没有对照官方测试用例或示例验证 | 直到集成测试才发现大量问题 |

## 五、经验教训

### 教训 1：先调研，再动手

> **在开始任何涉及外部协议/标准/规范的实现之前，必须先回答三个问题：**
> 1. 官方有没有提供 SDK？
> 2. 有没有官方示例代码？
> 3. 社区有没有成熟的实现可以参考？

如果当时先花了半天调研，就能避免 17 天的返工。

### 教训 2：AI 不会替你做技术选型

AI 能根据你的描述生成代码，但：
- AI **不会主动告诉你**"其实有官方 SDK 可以直接用"
- AI **不会质疑你的方案**是否是最优解
- AI 会忠实地按照你的指令去做，哪怕方向是错的

**技术选型和方案设计是人的责任，不是 AI 的。**

### 教训 3：对不了解的领域，更要小心

- 对自己熟悉的领域（如 Spring Boot 开发），AI 的输出质量很高
- 对不了解的领域（如 A2A 协议），无法判断 AI 输出的正确性
- **越是不了解的领域，越需要先建立认知，再让 AI 辅助**

### 教训 4：协议类代码不适合"先有再优化"

不同于业务代码可以快速迭代，协议实现要求**一次性正确**：
- 数据模型必须与规范完全一致
- 错误码必须符合标准
- 端点路径必须精确匹配
- 序列化/反序列化格式必须合规

这类代码应该基于官方 SDK 或经过验证的参考实现来写，而不是让 AI 从零生成。

## 六、如果重来会怎么做

### 正确的做法

```
1. 花 2 小时阅读 A2A 官方 GitHub 仓库
   → 发现官方有 Java SDK
   → 发现 SDK 有 Spring WebFlux 示例

2. 花 1 天学习 SDK 的核心接口
   → AgentExecutor（必须实现的核心接口）
   → RestHandler（REST 传输处理器）
   → TaskStore（任务存储）

3. 花 1-2 天基于 SDK 做适配
   → 实现 AgentExecutor 接口
   → 配置 ShenYu 网关集成
   → 参考官方示例调通

4. 总计 3-4 天，而不是 17 天
```

### 通用方法论

对于任何涉及外部协议/标准的开发：

```
调研（官方文档 + SDK + 社区方案）
  → 选型（用官方 SDK 还是社区方案）
    → 学习（核心接口 + 示例代码）
      → 适配（在项目中集成）
        → 验证（对照规范测试）
```
